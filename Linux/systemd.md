# Systemd

## 1. Зачем нужен Systemd?

Вы написали крутого Telegram-бота, вы заходите на сервер по SSH, запускаете `node bot.js`, бот работает, но как только вы закрываете терминал или у вас моргает интернет — **бот умирает**.

Вам нужно, чтобы программа:
1. Работала в фоне (независимо от вашей SSH-сессии).
2. **Сама перезапускалась**, если код упал с ошибкой.
3. Автоматически запускалась **при перезагрузке сервера** (если хостер перезагрузил машину).
4. Куда-то сохраняла свои логи (`console.log` или `print`).

В Linux за всё это отвечает **Systemd** (System and Service Manager), любая фоновая программа под управлением Systemd называется **демоном (daemon)**.

---

## 2. Unit

Чтобы Systemd узнал о вашем приложении, нужно создать специальный конфигурационный файл (unit-файл). 

Перейдем в системную папку и создадим файл с расширением `.service` (потребуются права `root` или `sudo`):
```bash
sudo nano /etc/systemd/system/mybot.service
```

Вставим туда следующую структуру (это золотой стандарт для 90% приложений):

```ini
[Unit]
Description=My Telegram Bot
# Сервис запустится только после того, как на сервере появится сеть
After=network.target

[Service]
# Полный путь до интерпретатора и полный путь до вашего скрипта
ExecStart=/usr/bin/node /opt/mybot/app.js
# Папка, из которой запускается скрипт
WorkingDirectory=/opt/mybot
# От имени какого пользователя запускать (не запускайте от root без необходимости!)
User=www-data
# Что делать, если скрипт упадет? always - всегда перезапускать
Restart=always
# Пауза перед перезапуском (чтобы не положить процессор при цикличных ошибках)
RestartSec=3

# Можно передавать переменные окружения
Environment=NODE_ENV=production
Environment=PORT=8080

[Install]
# Запускать в обычном многопользовательском режиме (без графического интерфейса)
WantedBy=multi-user.target
```

## 3. Основные команды (systemctl)

После того как вы создали или изменили файл `.service`, **обязательно** скажите Systemd перечитать конфигурации:
```bash
sudo systemctl daemon-reload
```

Теперь вы можете управлять своим приложением с помощью команды `systemctl`:

*   `sudo systemctl start mybot` — **Запустить** приложение.
*   `sudo systemctl stop mybot` — **Остановить** приложение.
*   `sudo systemctl restart mybot` — **Перезапустить** (удобно при деплое нового кода).
*   `sudo systemctl status mybot` — Посмотреть **статус** (работает ли, сколько потребляет памяти, последняя ошибка).

### Автозагрузка при старте сервера
Команда `start` запускает приложение только сейчас. Если сервер перезагрузится, бот не встанет. Чтобы включить автозагрузку, выполните:
```bash
sudo systemctl enable mybot
```
*(Чтобы выключить автозагрузку: `systemctl disable mybot`).*

## 4. Куда делись логи? (journalctl)

Systemd перехватывает всё, что ваше приложение выводит в консоль (stdout и stderr), и сохраняет в свой бинарный журнал.

Для чтения логов используется утилита **journalctl**:

*   `sudo journalctl -u mybot` — Показать все логи сервиса `mybot`.
*   `sudo journalctl -u mybot -f` — **(Самая частая команда)** Показать логи в реальном времени (режим "слежки", аналог `tail -f`).
*   `sudo journalctl -u mybot --since "1 hour ago"` — Показать логи только за последний час.
*   `sudo journalctl -u mybot -e` — Сразу промотать в самый конец журнала (к свежим ошибкам).

---

### Резюме
*   **Systemd** — инструмент Linux для управления фоновыми процессами.
*   Файлы конфигураций (`.service`) лежат в `/etc/systemd/system/`.
*   Управление сервисом: `systemctl start / stop / restart / status`.
*   Включение автозапуска: `systemctl enable`.
*   После любого изменения файла `.service` обязательно делайте `systemctl daemon-reload`.
*   Просмотр логов: `journalctl -u имя_сервиса -f`.
