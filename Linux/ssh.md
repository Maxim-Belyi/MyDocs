# SSH и SSH-ключи

## 1. SSH (Secure Shell) — сетевой протокол, который позволяет управлять удаленным компьютером через командную строку. 

В отличие от старых протоколов (например, Telnet), SSH шифрует весь трафик, даже если хакер перехватит данные между вашим ноутбуком и сервером (например, когда вы сидите в публичном Wi-Fi), он увидит только бессмысленный набор символов и не сможет украсть ваши пароли.

Обычно SSH работает на **22 порту**.

---

## 2. Как работают SSH-ключи? (Публичный и Приватный)

Вы можете входить на сервер по обычному паролю, но это небезопасно (его могут подобрать боты) и неудобно (приходится каждый раз его вводить). Стандарт индустрии — вход по **SSH-ключам** (асимметричная криптография).

При генерации создаются сразу **два** файла (пара):

1.  **Приватный ключ (Private Key):** Лежит только на вашем компьютере (обычно в папке `~/.ssh/`). Это ваш "паспорт", его **КАТЕГОРИЧЕСКИ НЕЛЬЗЯ** никому передавать, пересылать в Telegram или пушить в Git.
2.  **Публичный ключ (Public Key):** Файл с расширением `.pub` - это "замок". Вы можете свободно копировать его на любые серверы, в GitHub, GitLab или отдавать админам. 

**Как происходит магия:**
Вы кладете свой публичный ключ (замок) на сервер. Когда вы пытаетесь подключиться, сервер "запирает" случайное сообщение вашим публичным ключом, если ваш компьютер может расшифровать это сообщение своим приватным ключом — сервер пускает вас без пароля.

---

## 3. Какие бывают ключи (Алгоритмы)?

Если вы введете старую команду `ssh-keygen`, она по умолчанию создаст ключ типа **RSA**. Это старый, надежный, но тяжеловесный стандарт.

Сегодня стандартом де-факто является алгоритм **Ed25519**.
*   Он намного короче (ключ помещается в одну строчку).
*   Он генерируется и работает быстрее.
*   Он математически более безопасен против современных атак.

> **Лучшая практика:** Всегда генерируйте новые ключи с алгоритмом Ed25519.

---

## 4. Создать ключ и закинуть его на сервер

### Шаг 1: Генерация ключа
Открываем терминал на своем компьютере и пишем:
```bash
ssh-keygen -t ed25519 -C "your_email@example.com"
```
*   `-t ed25519` — выбираем современный алгоритм.
*   `-C "..."` — комментарий (обычно пишут свой email или название компьютера, чтобы потом вспомнить, чей это ключ).

Система спросит, куда сохранить ключ (жмите `Enter` для пути по умолчанию `~/.ssh/id_ed25519`) и предложит задать пароль на сам ключ (passphrase). Для локальной разработки пароль можно оставить пустым (снова два раза `Enter`).

### Шаг 2: Копирование на сервер
Теперь нам нужно закинуть публичный ключ (`.pub`) на сервер. Для этого в Linux есть специальная утилита:
```bash
ssh-copy-id username@192.168.1.10
```
Она попросит ввести пароль от сервера один (последний) раз. 

*Что происходит под капотом?* Эта команда просто берет текст из вашего файла `id_ed25519.pub` и дописывает его на сервере в специальный файл: `~/.ssh/authorized_keys`.

Всё! Теперь команда `ssh username@192.168.1.10` пустит вас мгновенно без пароля.

## 5. Магия `~/.ssh/config` (Как не писать длинные команды)

У вас на работе может быть сложный сервер: логин длинный, IP-адрес не запомнить, порт нестандартный (например, 2222), и нужен отдельный ключ.
Каждый раз писать `ssh super_admin@185.123.45.67 -p 2222 -i ~/.ssh/work_key` — это мучение.

Решение — создать локальный конфигурационный файл.
Откройте (или создайте) файл `~/.ssh/config` на своем компьютере и напишите:

```text
Host prod
    HostName 185.123.45.67
    User super_admin
    Port 2222
    IdentityFile ~/.ssh/work_key
```

Теперь, чтобы подключиться к этому серверу (или отправить на него файлы через `rsync`), вам достаточно написать:
```bash
ssh prod
```
SSH сам подставит нужный IP, пользователя, порт и ключ. Это экономит массу времени!

---

### Резюме
*   **SSH** — зашифрованный протокол для управления серверами (обычно порт 22).
*   **Приватный ключ (`id_ed25519`)** — прячем от всех. **Публичный (`id_ed25519.pub`)** — отдаем серверам.
*   Современный стандарт ключа — **Ed25519**. Команда для создания: `ssh-keygen -t ed25519`.
*   Быстрый заброс ключа на сервер: `ssh-copy-id user@host`.
*   Используйте файл `~/.ssh/config`, чтобы давать серверам короткие имена (алиасы) и не запоминать IP-адреса.
