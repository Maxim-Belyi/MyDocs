# Чистая Архитектура в Go (на примере шаблона Evrone)

## 1. Чистая архитектура (Clean Architecture) Роберта Мартина (Uncle Bob) — это способ организации кода, при котором **бизнес-логика приложения полностью изолирована от внешнего мира** (баз данных, фреймворков, HTTP, gRPC и очередей сообщений).

В Go эталоном реализации этого подхода считается open-source проект [evrone/go-clean-template](https://github.com/evrone/go-clean-template).

**Главное правило (Dependency Rule):** Зависимости направлены только внутрь. Внешние слои знают о внутренних, но внутренние ничего не знают о внешних.

```text
[Транспорт (HTTP/gRPC)] → [Бизнес-логика (UseCase)] → [База данных (Repository)]
                                        ↑
                   [Ядро (Entity)] — не зависит вообще ни от чего
```
*Если завтра мы решим заменить PostgreSQL на MongoDB, а REST API на gRPC — ядро и бизнес-логика приложения не изменятся ни на одну строчку кода.*

---

## 2. Структура директорий (Standard Go Layout)

Проект использует стандартную раскладку папок, принятую в Go-сообществе:

*   `/cmd/app/` — Точка входа в программу (`main.go`).
*   `/internal/` — Приватный код приложения (недоступен для импорта из других проектов). Здесь лежит вся архитектура.
*   `/pkg/` — Публичные, переиспользуемые пакеты (например, обертки над логгером, подключением к БД или RabbitMQ).
*   `/docs/` — Swagger документация и `.proto` файлы.

## 3. Разбор слоев архитектуры (Снизу вверх)

Архитектура состоит из 5 основных блоков. Рассмотрим их от самого независимого (центра) к самым зависимым (краям).

### Слой 1: Entity (Ядро и Сущности)
**Путь:** `internal/entity/`

Здесь описываются главные объекты бизнеса (например, `Task`, `User`) и доменные ошибки.
> **(Rich Domain Model):** В хорошей архитектуре сущность — это не просто структура с данными, но и методы, управляющие ее состоянием. 

Пример машины состояний прямо внутри сущности `Task`:
```go
// entity/task.go
func (t *Task) Transition(newStatus TaskStatus) error {
    validTransitions := map[TaskStatus][]TaskStatus{
        TaskStatusTodo:       {TaskStatusInProgress},
        TaskStatusInProgress: {TaskStatusDone, TaskStatusTodo},
        TaskStatusDone:       {},  // из Done перейти никуда нельзя
    }
    // Логика проверки...
}
```
Также здесь лежат **доменные ошибки**:
```go
var ErrTaskNotFound = errors.New("task not found")
```
*(Сущности ничего не знают про HTTP-статус 404, это просто ошибка `ErrTaskNotFound`)*.

### Слой 2: UseCase (Бизнес-логика)
**Путь:** `internal/usecase/`

Здесь живут сценарии использования. UseCase принимает запросы от пользователя, проверяет бизнес-правила и командует базой данных,
именно здесь применяется **Принцип Инверсии Зависимостей (DIP)**, UseCase зависит не от конкретной базы данных, а от **интерфейса**.

```go
// internal/usecase/contracts.go
// Контракт: Мне нужен кто-то, кто умеет сохранять задачи. Кто это будет - мне неважно.
type TaskRepo interface {
    Store(ctx context.Context, task *entity.Task) error
}

// internal/usecase/task/task.go
type UseCase struct {
    repo usecase.TaskRepo // Внедряем зависимость через интерфейс
}
```
Также UseCase оборачивает ошибки, добавляя контекст (trace), используя `%w`:
```go
return fmt.Errorf("TaskUseCase - Create - uc.repo.Store: %w", err)
```

### Слой 3: Repository (Адаптеры данных)
**Путь:** `internal/repo/`

Здесь лежат конкретные реализации интерфейсов из UseCase, например, `repo/persistent/task_postgres.go`.
*   Вместо тяжелых ORM (вроде GORM) в шаблоне используется **Query Builder** (например, `squirrel`), чтобы писать безопасные SQL-запросы без потери контроля над БД.
*   Репозиторий — это не только база данных, здесь же может лежать папка `webapi/`, которая ходит по HTTP в сторонний сервис (например, сервис переводов), для UseCase нет разницы, откуда репозиторий берет данные.

### Слой 4: Controller (Транспортный слой)
**Путь:** `internal/controller/`

Слой, который принимает запросы из внешнего мира (HTTP, gRPC, RabbitMQ).
Его задачи:
1. Распарсить JSON / Protobuf / Message.
2. Вызвать метод интерфейса `UseCase`.
3. Конвертировать ответ и **транслировать доменные ошибки в транспортные** (например, `entity.ErrTaskNotFound` превратить в HTTP `404 Not Found`).

### Слой 5: App (Composition Root / Точка сборки)
**Путь:** `internal/app/app.go`

В Go не принято использовать "магические" фреймворки для Dependency Injection (DI контейнеры). Сборка происходит вручную в одном месте:
```go
func Run(cfg *config.Config) {
    // 1. Создаем инфраструктуру (Подключение к БД)
    pg := postgres.New(cfg.PG.URL)

    // 2. Создаем UseCase, прокидывая в него интерфейс БД
    taskUC := task.New(pg)

    // 3. Создаем HTTP-сервер, прокидывая в него интерфейс UseCase
    restapi.NewRouter(handler, taskUC)
    
    // 4. Запускаем серверы и ждем сигнала остановки
}
```

## 4. Инфраструктурные паттерны

Что еще делает этот шаблон эталонным:

1.  **Graceful Shutdown (Плавная остановка):** Приложение слушает сигналы операционной системы (`SIGTERM`, `SIGINT`). При попытке остановить контейнер, приложение перестает принимать новые запросы, дорабатывает текущие и корректно закрывает соединения с БД.
2.  **Конфигурация через Environment (12-Factor App):** Конфиг парсится строго из переменных окружения в Go-структуру (через теги `env:"HTTP_PORT"`). Никаких хардкод `.yaml` файлов в логике.
3.  **Context Propagation:** Объект `context.Context` передается первым аргументом **во все** методы сквозь все слои. Это позволяет отменять долгие запросы к БД, если клиент закрыл браузер по таймауту.

## 5. Чего намеренно НЕТ в проекте?

На собеседованиях любят спрашивать, почему вы не используете ту или иную технологию, создатели шаблона намеренно отказались от:
*   **Глобальных переменных:** Никаких глобальных логгеров или подключений к БД. Всё передается через DI.
*   **Функций `init()`:** Они создают непредсказуемые побочные эффекты до старта функции `main`.
*   **ORM (GORM):** Слишком много магии и проблем с производительностью на сложных запросах.
*   **Использования `panic` в бизнес-логике:** Ошибки возвращаются как `error` (Errors are Values) и обрабатываются на уровне контроллера.

### Резюме: Путь одного запроса

Как проходит запрос на создание задачи:
1. **[Controller] HTTP POST /tasks** $\rightarrow$ Валидирует JSON, вызывает UseCase.
2. **[UseCase] Create()** $\rightarrow$ Присваивает статус `Todo`, валидирует бизнес-правила, вызывает Репозиторий.
3. **[Repository] Store()** $\rightarrow$ Генерирует SQL-запрос `INSERT INTO tasks...`, пишет в PostgreSQL.
4. **[Возврат]** $\rightarrow$ Репозиторий возвращает сущность в UseCase $\rightarrow$ UseCase возвращает в Controller $\rightarrow$ Controller отдает HTTP 201 Created.
