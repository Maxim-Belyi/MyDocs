Rate Limiting (Ограничение скорости)

Rate Limiter — это механизм, который ограничивает количество запросов, которые пользователь или сервис может сделать к вашей системе за единицу времени. Это абсолютно критический паттерн для защиты публичных API от брутфорса, DDoS-атак и просто неаккуратных ботов.

На собеседованиях по System Design и написанию кода на Go часто просят реализовать простейший Rate Limiter.

1. Алгоритм Token Bucket (Маркерная корзина)​

Самый популярный алгоритм ограничения скорости.

  • У нас есть "корзина", в которой может храниться максимум N токенов (маркеров).
  • Каждую секунду в корзину с постоянной скоростью добавляется R новых токенов.
  • Если корзина полна, новые токены выбрасываются.
  • Когда приходит запрос, он должен забрать 1 токен из корзины. Если токенов нет — запрос отклоняется (429 Too Many Requests).

Плюсы Token Bucket: Позволяет обрабатывать кратковременные всплески трафика (bursts). Если клиент долго молчал, его корзина наполнилась, и он может сделать сразу N запросов подряд.

Реализация на Go (Стандартная библиотека)​

В Go есть готовая, высокопроизводительная и потокобезопасная реализация Token Bucket в пакете golang.org/x/time/rate.

package main

import (
	"fmt"
	"net/http"
	"golang.org/x/time/rate"
)

// rate.Limit(2) — восстанавливаем 2 токена в секунду
// burst = 5 — максимальный размер корзины (всплеск)
var limiter = rate.NewLimiter(rate.Limit(2), 5)

func limitMiddleware(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		// limiter.Allow() забирает 1 токен. 
		// Если токенов нет, возвращает false.
		if !limiter.Allow() {
			http.Error(w, "Too Many Requests", http.StatusTooManyRequests)
			return
		}
		
		next.ServeHTTP(w, r)
	})
}

2. Реализация Token Bucket "руками" (через каналы)​

На алгоритмической секции вас могут попросить написать лимитер без внешних библиотек. В Go это элегантно делается через буферизованные каналы и тикеры.

package main

import (
	"fmt"
	"time"
)

func main() {
	// Корзина вмещает максимум 5 токенов
	burst := 5
	bucket := make(chan struct{}, burst)

	// Изначально наполняем корзину до краев
	for i := 0; i < burst; i++ {
		bucket <- struct{}{}
	}

	// Горутина, которая пополняет корзину (по 1 токену каждые 500мс)
	go func() {
		ticker := time.NewTicker(500 * time.Millisecond)
		defer ticker.Stop()
		for range ticker.C {
			select {
			case bucket <- struct{}{}:
				// Токен успешно добавлен
			default:
				// Корзина полна, токен сгорает (канал переполнен)
			}
		}
	}()

	// Симуляция приходящих запросов
	for i := 1; i <= 10; i++ {
		select {
		case <-bucket:
			fmt.Printf("Запрос %d обработан\n", i)
		default:
			fmt.Printf("Запрос %d ОТКЛОНЕН (429 Too Many Requests)\n", i)
		}
		// Запросы приходят очень быстро
		time.Sleep(100 * time.Millisecond)
	}
}

3. Распределенный Rate Limiter​

Если ваше приложение крутится в 10 подах Kubernetes, локальный Rate Limiter в памяти (как rate.NewLimiter) будет ограничивать трафик только для одного инстанса. Пользователь может сделать в 10 раз больше запросов, чем разрешено, если балансировщик раскидает запросы по разным подам.

Решение для продакшена: Использовать Redis. Существуют готовые алгоритмы реализации Token Bucket или Sliding Window с использованием Lua-скриптов в Redis, чтобы проверка и списание токенов были атомарными для всех подов одновременно.
