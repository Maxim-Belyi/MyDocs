Worker Pool (Пул воркеров)

Worker Pool (Пул обработчиков) — это, пожалуй, самый важный и часто используемый паттерн конкурентности в Go.

1. Зачем это нужно?​

В Go очень легко запускать горутины (go doWork()). Если вам нужно обработать 1 миллион картинок, вы можете запустить 1 миллион горутин. Но это плохая идея:

  1. Вы исчерпаете оперативную память (каждая горутина весит около 2 КБ).
  2. Вы создадите огромную нагрузку на CPU из-за постоянного переключения контекста (context switching).
  3. Если эти горутины делают запросы в БД или по сети, они моментально положат БД (too many connections) или исчерпают лимит открытых файлов/сокетов в ОС.

Решение: Создать фиксированное количество горутин (воркеров), например 100 штук. Они будут брать задачи из общей очереди (канала), обрабатывать их и брать следующие. Так мы контролируем потребление ресурсов.

2. Реализация на Go​

Классический паттерн требует трех вещей:

  1. Канал для задач (jobs).
  2. Канал для результатов (results).
  3. sync.WaitGroup, чтобы дождаться завершения всех воркеров перед закрытием канала результатов.
package main

import (
	"fmt"
	"sync"
	"time"
)

// worker — функция, которую будут выполнять наши горутины.
// Она читает из канала jobs и пишет в канал results.
func worker(id int, jobs <-chan int, results chan<- int, wg *sync.WaitGroup) {
	defer wg.Done() // Обязательно сообщаем, что воркер закончил работу
	
	// Воркер блокируется на чтении из канала.
	// Цикл range автоматически завершится, когда канал jobs будет закрыт.
	for j := range jobs {
		fmt.Printf("Воркер %d начал задачу %d\n", id, j)
		time.Sleep(time.Second) // Симуляция тяжелой работы
		fmt.Printf("Воркер %d завершил задачу %d\n", id, j)
		
		results <- j * 2 // Отправляем результат
	}
}

func main() {
	const numJobs = 5
	const numWorkers = 3

	// Буферизованные каналы (размер буфера равен количеству задач, 
	// чтобы генератор задач не блокировался)
	jobs := make(chan int, numJobs)
	results := make(chan int, numJobs)

	var wg sync.WaitGroup

	// 1. Запускаем воркеров (они сразу уснут, так как канал jobs пуст)
	for w := 1; w <= numWorkers; w++ {
		wg.Add(1)
		go worker(w, jobs, results, &wg)
	}

	// 2. Отправляем задачи в канал
	for j := 1; j <= numJobs; j++ {
		jobs <- j
	}
	
	// 3. Закрываем канал задач.
	// Это сигнал воркерам: "Новых задач больше не будет. 
	// Доделайте то, что в буфере канала, и завершайтесь."
	close(jobs)

	// 4. Ожидаем завершения всех воркеров в отдельной горутине, 
	// чтобы можно было закрыть канал результатов
	go func() {
		wg.Wait()
		close(results)
	}()

	// 5. Читаем результаты.
	// Если бы мы не закрыли канал results выше, этот цикл заблокировался бы навсегда (deadlock)
	for a := range results {
		fmt.Println("Результат:", a)
	}
}

Итог для собеседования​

На собеседовании вас попросят написать этот код на доске (или в блокноте). Главные моменты, на которые смотрит интервьюер:

  1. Понимаете ли вы, что канал jobs нужно закрывать со стороны отправителя ("sender closes channel").
  2. Используете ли вы sync.WaitGroup для ожидания воркеров.
  3. Понимаете ли вы, где вызвать close(results). Ошибка новичка — вызов wg.Wait() и close(results) в основном потоке перед чтением результатов. Это приведет к дэдлоку, так как никто не будет читать результаты, пока wg.Wait() заблокирован. Поэтому wg.Wait() выносят в отдельную горутину.
