Circuit Breaker (Предохранитель)

Circuit Breaker (Предохранитель) — это паттерн обеспечения стабильности распределенных систем.

Представьте, что ваш микросервис А обращается к микросервису Б (например, к платежному шлюзу). Если сервис Б "упал" и не отвечает (таймаут 10 секунд), а к сервису А приходит 100 запросов в секунду, то очень скоро у сервиса А исчерпаются все свободные потоки (горутины), сокеты и память. Сервис А упадет вслед за сервисом Б. Это называется каскадным сбоем.

Circuit Breaker предотвращает каскадные сбои, отслеживая ошибки и, при достижении критического порога, "разрывая цепь", чтобы сервис А сразу отдавал ошибку клиентам (fail fast), даже не пытаясь стучаться в лежащий сервис Б.

1. Три состояния предохранителя​

  1. Closed (Закрыт): Нормальное состояние. Ток течет, запросы отправляются к сервису Б. Предохранитель считает ошибки.
  2. Open (Открыт): Ошибок стало слишком много (перегрев). Предохранитель разрывает цепь. Все последующие запросы моментально отклоняются с ошибкой (например, ErrCircuitOpen), без обращения к сети.
  3. Half-Open (Полуоткрыт): Спустя время (таймаут охлаждения) предохранитель пропускает один тестовый запрос. Если он успешен, цепь замыкается (Closed). Если нет — цепь снова обрывается (Open).

2. Использование gobreaker в Go​

Для реализации этого паттерна в Go стандартом де-факто является библиотека github.com/sony/gobreaker (разработана в Sony).

package main

import (
	"errors"
	"fmt"
	"time"
	"github.com/sony/gobreaker"
)

var cb *gobreaker.CircuitBreaker

func init() {
	st := gobreaker.Settings{
		Name:        "HTTPClient",
		MaxRequests: 1,                // Сколько тестовых запросов пропускать в состоянии Half-Open
		Interval:    10 * time.Second, // Период сброса счетчиков в состоянии Closed
		Timeout:     5 * time.Second,  // Время нахождения в состоянии Open до перехода в Half-Open
		ReadyToTrip: func(counts gobreaker.Counts) bool {
			// Логика перехода в Open: если процент ошибок больше 50%
			// и было сделано хотя бы 10 запросов
			failureRatio := float64(counts.TotalFailures) / float64(counts.Requests)
			return counts.Requests >= 10 && failureRatio >= 0.5
		},
		OnStateChange: func(name string, from gobreaker.State, to gobreaker.State) {
			fmt.Printf("Circuit Breaker '%s' changed state: %s -> %s\n", name, from, to)
		},
	}
	cb = gobreaker.NewCircuitBreaker(st)
}

func main() {
	// Выполняем опасный сетевой вызов через Circuit Breaker
	result, err := cb.Execute(func() (interface{}, error) {
		// Симуляция обращения к стороннему API
		// resp, err := http.Get("https://api.example.com")
		// ...
		
		// Для примера симулируем ошибку
		return nil, errors.New("network timeout")
	})

	if err != nil {
		// Если предохранитель открыт, вернется gobreaker.ErrOpenState
		if err == gobreaker.ErrOpenState {
			fmt.Println("Fast failure: Circuit is OPEN, not calling network")
		} else {
			fmt.Println("Request failed:", err)
		}
		return
	}

	fmt.Println("Success:", result)
}

Итог для собеседования​

  • Если спрашивают, как защитить систему от медленного/упавшего зависимого сервиса — отвечайте Circuit Breaker.
  • Если спрашивают про ретраи (как достучаться до флапающего сервиса) — отвечайте Exponential Backoff.
  • Часто эти паттерны комбинируют: Circuit Breaker защищает от каскадных сбоев, а Backoff пытается доставить сообщение, если оно не было отклонено Breaker'ом.
