# Брокер сообщений RabbitMQ: Архитектура, доставка и надежность

RabbitMQ — это брокер сообщений, работающий по стандарту AMQP (Advanced Message Queuing Protocol). Он используется в распределенных системах для асинхронного взаимодействия микросервисов, маршрутизации сообщений и обеспечения гарантированной доставки задач между отправителями (Producer) и получателями (Consumer).

---

## 1. Отправка сообщений (Producer -> RabbitMQ)

На уровне протокола AMQP 0-9-1 базовая операция публикации (`basic.publish`) отправляет сообщения поштучно. Приложение открывает TCP-соединение с брокером, создает внутри него логический канал (Channel) и передает данные отдельными фреймами.

Прямая отправка массива сообщений одной командой в стандарте AMQP 0-9-1 не предусмотрена. Для повышения пропускной способности при отправке применяются два основных механизма:

*   **TCP-буферизация:** На уровне сетевого стека множество последовательных вызовов `basic.publish` могут объединяться в один физический TCP-пакет.
*   **Publisher Confirms (Асинхронные подтверждения):** Чтобы убедиться в сохранении данных на диск брокера без синхронного ожидания ответа на каждое сообщение, отправитель переводит канал в режим подтверждений. Сообщений отправляются непрерывным потоком, а RabbitMQ периодически возвращает пакетное подтверждение (`Ack`) с указанием номера `delivery_tag`, подтверждая запись всех сообщений вплоть до этого номера.

---

## 2. Доставка сообщений (RabbitMQ -> Consumer)

В отличие от систем с Pull-моделью (например, Apache Kafka, где получатель сам опрашивает сервер), RabbitMQ работает по **Push-модели**. Как только сообщение попадает в очередь, брокер инициирует его передачу по открытому TCP-соединению в сторону консьюмера.

Физически по сети сообщения всегда передаются как отдельные пакеты. Скорость и объем выдачи регулируются механизмом **QoS (Quality of Service)** через параметр `prefetch_count`.

### Работа параметра prefetch_count
`prefetch_count` определяет максимальное количество неподтвержденных сообщений, которое брокер может отправить в канал консьюмера:

*   **`prefetch_count = 1`:** Брокер отправляет ровно одно сообщение и блокирует дальнейшую выдачу для этого канала, пока не получит подтверждение (`ack`). Этот режим применяется для синхронной и последовательной обработки задач, где важна равномерная загрузка воркеров.
*   **`prefetch_count > 1` (например, 50):** Брокер последовательно отправляет до 50 сообщений в буфер памяти приложения, не дожидаясь промежуточных подтверждений. Этот подход используется в многопоточных или асинхронных приложениях (например, на Go), где пачка входящих сообщений распределяется по пулу горутин для параллельного выполнения.

## 3. Механика подтверждений: Ack, Nack и Reject

После того как RabbitMQ отправил сообщение консьюмеру, оно переходит в статус **Unacked**. В этом состоянии сообщение остается в оперативной памяти и на диске брокера, но становится недоступным для других параллельных воркеров.

Для завершения цикла обработки консьюмер обязан отправить в брокер один из трех ответов:

### `basic.ack` (Acknowledge)
Положительное подтверждение успешной обработки. Получив `ack`, RabbitMQ окончательно удаляет сообщение из очереди.

### `basic.reject`
Отказ от обработки одного конкретного сообщения. Поведение брокера зависит от флага `requeue`:
*   `requeue = true`: Сообщение возвращается в начало или конец очереди и немедленно доставляется повторно (тому же или другому консьюмеру). Если ошибка вызвана некорректными данными в самом сообщении, это может привести к бесконечному циклу обработки.
*   `requeue = false`: Сообщение удаляется из основной очереди. Если для очереди настроен обменник «мертвых писем» (**Dead Letter Exchange / DLX**), сообщение перенаправляется туда для анализа или отложенной обработки.

### `basic.nack` (Negative Acknowledge)
Расширение протокола, добавленное разработчиками RabbitMQ. В отличие от `basic.reject`, команда `nack` поддерживает флаг `multiple`:
*   При `multiple = true` брокер отменяет (или перенаправляет в DLX при `requeue = false`) все неподтвержденные сообщения в канале вплоть до указанного `delivery_tag`. Это позволяет сбросить сразу целую пачку задач при возникновении критической ошибки (например, при недоступности базы данных).

### Аварийное завершение консьюмера
Если процесс консьюмера завершается аварийно (сбой по Out of Memory, `kill -9` или обрыв сетевого соединения) без отправки `ack`/`nack`, TCP-соединение закрывается. RabbitMQ автоматически снимает статус `Unacked` со всех сообщений этого канала, возвращает их в состояние `Ready` и перераспределяет между активными консьюмерами. При этом сообщениям выставляется системный флаг `redelivered: true`.

## 4. Идемпотентность и предотвращение дубликатов

В распределенных системах RabbitMQ обеспечивает модель гарантии доставки **At-Least-Once** (хотя бы один раз). Из-за сетевых сбоев или перезапусков возможны ситуации повторной доставки (когда `redelivered = true`).

### Пример проблемы дублирования
1. Консьюмер получает сообщение, выполняет бизнес-логику и успешно записывает данные в базу данных.
2. Процесс аварийно завершается до того, как успевает отправить `basic.ack` по сети в RabbitMQ.
3. Брокер считает сообщение необработанным и возвращает его в очередь.
4. Другой воркер получает то же самое сообщение и повторно выполняет запись в базу данных.

### Три архитектурных правила для обеспечения идемпотентности

#### 1. Подтверждение строго после транзакции в БД
Отправка команды `basic.ack` должна происходить только после фиксации транзакции в базе данных:
```go
// 1. Открываем транзакцию
tx, err := db.BeginTx(ctx, nil)

// 2. Выполняем SQL-запросы
_, err = tx.ExecContext(ctx, query, args...)

// 3. Фиксируем транзакцию на диске БД
err = tx.Commit()

// 4. Отправляем подтверждение в RabbitMQ только после успешного Commit
_ = channel.Ack(msg.DeliveryTag, false)
```

#### 2. Уникальный идентификатор и ограничения на уровне БД
Поскольку сбой может произойти в микросекундном интервале между вызовом `Commit()` и отправкой `Ack()`, база данных должна быть финальным гарантом уникальности. 

Каждое сообщение должно содержать стабильный `message_id`. В таблице БД создается уникальный индекс (`UNIQUE INDEX`) по этой колонке, а вставка выполняется с обработкой конфликтов:
```sql
INSERT INTO orders (message_id, order_data) 
VALUES ($1, $2) 
ON CONFLICT (message_id) DO NOTHING;
```
Если сообщение придет повторно, база данных проигнорирует дубликат без вызова ошибки, после чего консьюмер сможет штатно отправить `ack`.

#### 3. Управление батчами при записи
При работе с пакетами (`prefetch_count > 1`) существуют два подхода к обработке:
*   **Поштучная обработка в цикле:** Каждое сообщение записывается отдельной транзакцией, после чего сразу отправляется индивидуальный `ack`. При сбое на 11-м сообщении из 100 первые 10 останутся подтвержденными, а оставшиеся 90 вернутся в очередь.
*   **Пакетная запись (Bulk Insert):** Все 100 сообщений записываются одним SQL-запросом, после чего отправляется один `ack` с флагом `multiple = true`. Этот способ снижает нагрузку на БД, но при ошибке в одной из строк весь пакет транзакции откатится, и все 100 сообщений будут доставлены заново.

## 5. Почему метка времени (Timestamp) не подходит для message_id

Использование текущего времени в качестве уникального ключа (`message_id`) приводит к коллизиям и потере данных по следующим причинам:

1.  **Скорость выполнения:** Современные процессоры выполняют тысячи операций в миллисекунду. Если воркер в цикле формирует события, десятки сообщений получат абсолютно одинаковую метку времени. Даже на уровне наносекунд (`time.Now().UnixNano()`) при параллельной работе нескольких горутин возможна генерация одинаковых значений из системных часов.
2.  **Горизонтальное масштабирование:** При работе нескольких независимых экземпляров сервиса на разных серверах вероятность того, что два разных узла сгенерируют события в одну и ту же долю секунды, существенно возрастает.
3.  **Рассинхронизация часов (NTP Drift):** Системное время на разных физических серверах подвержено рассинхронизации и периодической корректировке протоколом NTP (включая скачки времени назад), что нарушает монотонность.
4.  **Повторные попытки (Retries):** Если при первой попытке отправки (`T1`) произошел сбой сети, и сервис инициирует повторную отправку через секунду, генерация нового идентификатора по текущему времени создаст метку `T2`. Брокер и база данных воспримут это как две разные бизнес-операции вместо повтора одной и той же.

### Современные форматы идентификаторов
Для совмещения быстрой сортировки в индексах баз данных и криптографической уникальности применяются гибридные форматы:

*   **ULID / UUIDv7:** Старшие байты идентификатора содержат временную метку (обеспечивая последовательную сортировку и предотвращая фрагментацию B-Tree индексов), а младшие байты заполняются криптографически стойким случайным числом.
*   **Snowflake ID:** 64-битный числовой идентификатор, объединяющий временную метку, уникальный номер физического сервера/инстанса и монотонно растущий локальный счетчик для коллизий внутри одной миллисекунды.

## Итог: Практические советы

1.  **Настраивайте `prefetch_count` под задачи:** Используйте `1` для тяжелых последовательных задач и значения больше `1` при параллельной обработке в памяти (например, через горутины в Go).
2.  **Не используйте `requeue = true` при валидационных ошибках:** Если сообщение не может быть распарсено, отправляйте `reject` с `requeue = false` и направляйте его в Dead Letter Exchange.
3.  **Идемпотентность обязательна:** Проектируйте консьюмеры с расчетом на то, что любое сообщение может прийти повторно с флагом `redelivered: true`.
4.  **Отделяйте генерацию `message_id` от транспорта:** Идентификатор должен формироваться на этапе инициации бизнес-действия (например, при создании заказа) и оставаться неизменным при всех сетевых повторах.
5.  **Выбирайте `ULID` или `UUIDv7` вместо `time.Now()`:** Это гарантирует отсутствие коллизий при горизонтальном масштабировании и сохраняет высокую производительность индексов БД.
