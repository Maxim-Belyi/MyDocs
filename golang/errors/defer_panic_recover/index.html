Defer, Panic и Recover под капотом

Эта троица управляет потоком выполнения в исключительных ситуациях. В отличие от других языков, где есть try-catch, Go использует механизм отложенных вызовов и раскрутки стека. Понимание того, как это устроено в рантайме, помогает писать предсказуемый и быстрый код.

1. Механизм Defer: Как это работает на самом деле​

defer откладывает выполнение функции до момента выхода из текущей функции (либо через return, либо при панике).

Главные правила:​

  1. LIFO (Last In, First Out): Вызовы defer выполняются в обратном порядке: последний объявленный выполнится первым.
  2. Аргументы вычисляются немедленно: Значения, передаваемые в функцию под defer, фиксируются в момент объявления, а не в момент выполнения.
  3. Defer может изменять возвращаемые значения: Если возвращаемая переменная именована, defer может изменить её значение перед тем, как функция фактически вернет управление.
func namedReturn() (result int) {
    defer func() {
        result++ // Изменяем именованный результат
    }()
    return 5 // Функция вернет 6
}

Внутреннее устройство (Go 1.14+):​

Раньше defer был медленным, так как каждый вызов создавал структуру в куче. Сейчас Go использует три вида оптимизации:

  • Open-coded defers: Если количество defer известно при компиляции и их немного, компилятор просто вставляет их вызовы перед каждым return. Это практически бесплатно.
  • Stack-allocated defers: Если нельзя использовать open-coded, Go пытается выделить структуру _defer на стеке.
  • Heap-allocated defers: Самый медленный вариант (например, внутри цикла).

2. Анатомия Panic​

Когда вызывается panic, текущая горутина прекращает обычное выполнение и начинает раскрутку стека (stack unwinding).

  1. Рантайм создает структуру _panic.
  2. Go начинает идти вверх по стеку вызовов.
  3. Для каждой функции в стеке вызываются все её defer-функции.
  4. Если ни один defer не вызвал recover, программа завершается с дампом всех стеков горутин.

Важно: panic в одной горутине убивает всё приложение, а не только эту горутину.

3. Recover: Спасательный круг​

recover — это встроенная функция, которая останавливает раскрутку стека. Она возвращает значение, переданное в panic.

  • recover работает только внутри defer.
  • После recover выполнение продолжается после функции, которая вызвала panic. Код внутри функции, где случилась паника, после строки с panic не выполняется.
func protected() {
    defer func() {
        if r := recover(); r != nil {
            fmt.Println("Восстановление после:", r)
        }
    }()
    
    fmt.Println("Старт")
    panic("Ошибка!")
    fmt.Println("Этот код никогда не выполнится")
}

4. Цепочка паник​

Что будет, если внутри defer во время паники случится ещё одна паника? Go поддерживает цепочку паник. Каждая новая паника "наслаивается" на предыдущую. recover всегда перехватывает последнюю панику.

defer func() {
    recover() // Перехватит "Паника 2"
}()
defer func() {
    panic("Паника 2")
}()
panic("Паника 1")

5. Ошибки проектирования​

Паника как исключение: Никогда не используйте panic/recover для обычной логики (как try/catch). Это считается антипаттерном в Go. Паника — это признак того, что программа зашла в состояние, из которого невозможно безопасно продолжать работу (например, индекс за границами массива или повреждение данных).

Забытый recover в горутине: Если вы запускаете горутину в HTTP-сервере, и она паникует — упадет весь сервер. Всегда ставьте defer recover() в "корневых" горутинах фоновых задач.

Итог для профи​

  1. Defer — это не магия. Это структуры _defer, связанные в список.
  2. Используйте именованные возвращаемые значения, если вам нужно изменить результат функции или залогировать ошибку из defer.
  3. Не вызывайте os.Exit(), если хотите, чтобы отработали ваши defer. os.Exit завершает процесс немедленно, игнорируя всё.
  4. Следите за аргументами в defer. Если вы передаете time.Since(start) прямо в аргументы, время зафиксируется в начале функции, а не в конце. Используйте замыкание: defer func() { fmt.Println(time.Since(start)) }().
