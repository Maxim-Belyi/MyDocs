# REST API

**REST (REpresentational State Transfer)** — это не протокол и не жесткий стандарт, а **архитектурный стиль**. Это набор правил о том, как программы должны общаться друг с другом по сети, используя возможности протокола HTTP.

Данные в REST обычно передаются в формате **JSON** (реже XML).

## 1. Ресурсы, а не действия (Правильные URL)

Главное правило REST: **URL должен указывать на сущность (существительное), а не на действие (глагол).** Действие определяется HTTP-методом.

❌ **Плохо (RPC-стиль, так делать в REST нельзя):**
*   `/getUsers`
*   `/createUser`
*   `/deleteUser?id=5`

✅ **Хорошо (REST-стиль):**
Вместо глаголов мы используем коллекции (во множественном числе) и иерархию:
*   `GET /users` — получить список всех пользователей.
*   `POST /users` — создать нового пользователя.
*   `GET /users/5` — получить данные пользователя с ID 5.
*   `GET /users/5/orders` — получить все заказы пользователя с ID 5.

---

## 2. HTTP-методы (CRUD)

REST API использует стандартные методы HTTP для выполнения операций CRUD (Create, Read, Update, Delete).

1.  **GET (Чтение):** Запросить данные. *Не должен ничего менять на сервере!*
2.  **POST (Создание):** Отправить данные для создания новой сущности.
3.  **PUT (Полное обновление):** Заменяет целевой ресурс **целиком**. Если вы передали в `PUT` только поле `name`, а поле `age` не передали, то `age` должно стереться (замениться на null/default).
4.  **PATCH (Частичное обновление):** Изменяет **только переданные поля**. Передали только `name` — изменилось только имя, остальные данные остались нетронутыми.
5.  **DELETE (Удаление):** Удаляет ресурс.

## 3. Безопасность и Идемпотентность 

Это фундамент, без которого нельзя спроектировать надежную систему.

### Безопасные методы (Safe Methods)
Метод называется безопасным, если он **только читает** данные и никак не изменяет состояние сервера.
*   **Безопасные:** `GET`, `OPTIONS`, `HEAD`.
*   **Небезопасные:** `POST`, `PUT`, `PATCH`, `DELETE`.

### Идемпотентные методы (Idempotent Methods)
Метод называется идемпотентным, если **многократное** выполнение одного и того же запроса дает **тот же результат на сервере**, что и однократное.

*   **`GET` — идемпотентен.** Сколько бы раз вы ни запрашивали профиль, данные на сервере не меняются.
*   **`PUT` — идемпотентен.** Если вы 10 раз отправите запрос "установить возраст = 25", возраст останется 25.
*   **`DELETE` — идемпотентен.** Если вы удалите пользователя, его не будет. Если вы повторите запрос еще 5 раз (даже если сервер вернет ошибку 404 Not Found), *состояние сервера не изменится* — пользователя там по-прежнему нет.
*   **`POST` — НЕ идемпотентен.** Если вы 5 раз отправите запрос на создание заказа, в базе появится 5 одинаковых заказов.

*(Именно поэтому при сетевых сбоях безопасно автоматически повторять (retry) GET и PUT запросы, но опасно повторять POST без специальных ключей идемпотентности).*

## 4. HTTP-статусы (Как правильно отвечать)

API должен общаться с клиентом не только через JSON, но и через правильные коды ответа. Возвращать `200 OK` вместе с `{"error": "User not found"}` — это антипаттерн!

**Коды:**

*   🟢 **2xx: Успех**
    *   `200 OK` — Запрос выполнен (стандарт для GET, PUT, PATCH).
    *   `201 Created` — Ресурс успешно создан (ответ на правильный POST).
    *   `204 No Content` — Успех, но серверу нечего вернуть (часто возвращают на DELETE).
*   🟡 **4xx: Ошибка клиента (Он отправил дичь)**
    *   `400 Bad Request` — Невалидный JSON, не хватает обязательных полей.
    *   `401 Unauthorized` — Вы не авторизованы (нет токена или он протух). *Аналогия: "Я не знаю, кто ты".*
    *   `403 Forbidden` — Вы авторизованы, но вам туда нельзя (нет прав администратора). *Аналогия: "Я знаю, кто ты, но не пущу".*
    *   `404 Not Found` — Ресурс не найден (URL не существует или сущности с таким ID нет).
*   🔴 **5xx: Ошибка сервера (Мы накосячили)**
    *   `500 Internal Server Error` — Необработанная ошибка в коде бэкенда (упала база, Null Pointer Exception).

## 5. Главный принцип REST: Stateless (Без сохранения состояния)

Еще один важный вопрос: *"Что значит Stateless в архитектуре REST?"*

Это означает, что сервер **не должен хранить никакой контекст (сессию) о клиенте** между запросами. 
Каждый запрос от клиента должен содержать **всю необходимую информацию** для его выполнения (например, токен авторизации в заголовке `Authorization: Bearer <token>`).

**Почему это важно? (Масштабируемость)**
Если бы сервер хранил сессии пользователей в своей оперативной памяти, то при добавлении второго сервера вам пришлось бы настраивать хитрую синхронизацию памяти между ними (Sticky Sessions). 
В Stateless-архитектуре запрос может обработать *абсолютно любой* из 100 ваших серверов, так как всё нужное (токен) уже лежит внутри самого запроса.

## 6. HATEOAS

HATEOAS (Hypermedia As The Engine Of Application State) — это финальный (третий) уровень зрелости REST по модели Ричардсона. 

Суть: Сервер вместе с данными возвращает **ссылки на действия**, которые клиент может сделать с этими данными дальше. Клиенту не нужно жестко прописывать логику (хардкодить URL-ы), он просто "переходит по ссылкам" из ответа.

*Пример ответа HATEOAS:*
```json
{
    "id": 123,
    "status": "created",
    "total": 500,
    "_links": {
        "self": { "href": "/orders/123" },
        "pay":  { "href": "/orders/123/pay" },
        "cancel": { "href": "/orders/123/cancel" }
    }
}
```
*На практике HATEOAS используют крайне редко.*
