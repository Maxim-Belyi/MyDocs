# gRPC и Protocol Buffers

Долгие годы стандартом общения между серверами был **REST API** (передача JSON-ок поверх протокола HTTP/1.1). Но с приходом эпохи микросервисов, когда один запрос от пользователя может вызывать цепочку из десятка внутренних запросов между серверами, REST стал "узким горлышком" из-за своей медлительности.

Ему на смену пришел **gRPC** (gRPC Remote Procedure Calls) — фреймворк от Google, созданный специально для невероятно быстрого общения между микросервисами.

## 1. Суть концепции RPC (Remote Procedure Call)
В REST мы мыслим **ресурсами** и HTTP-методами: `GET /users/1` (Получить пользователя).
В RPC мы мыслим **функциями (процедурами)**: `GetUser(id=1)`. 
Идея RPC в том, чтобы вызывать функцию на удаленном сервере так же просто, как если бы это была обычная локальная функция внутри твоего кода.

---

## 2. Три кита gRPC 

gRPC достигает высокой производительности за счет двух главных технологий: **Protobuf** и **HTTP/2**.

### Кит 1: Protocol Buffers (Protobuf)
В REST данные передаются в текстовом формате JSON. JSON легко читать людям, но для компьютера он тяжелый: его нужно долго парсить, а текстовые ключи (типа `"first_name": "Alex"`) занимают много лишних байт в сети.

**Protobuf** — это бинарный формат сериализации данных.
1. Вы описываете "контракт" (структуры данных и функции) в специальном файле `api.proto`.
2. Компилятор `protoc` автоматически генерирует готовый Go-код (структуры и интерфейсы) из этого файла.
3. При отправке по сети данные сжимаются в бинарный код. Вместо `"first_name"` передается просто номер поля `1`. Это делает сообщения в 3–10 раз меньше по объему, а парсинг — в десятки раз быстрее, чем у JSON.

### Кит 2: HTTP/2
REST обычно работает поверх HTTP/1.1. gRPC работает **только** поверх HTTP/2.
Что это дает:
* **Мультиплексирование:** В HTTP/1.1 для 10 параллельных запросов нужно открыть 10 тяжелых TCP-соединений. В HTTP/2 все запросы и ответы асинхронно летают по **одному** TCP-соединению.
* **Сжатие заголовков (HPACK):** В HTTP/1.1 заголовки передаются простым текстом в каждом запросе (иногда они весят больше самих данных). HTTP/2 сжимает их.

### Кит 3: Строгая типизация и кодогенерация
Больше не нужно писать руками HTTP-клиенты, роутеры и парсить JSON. Вы пишете `.proto` файл, и gRPC генерирует готовые методы. Если клиент написан на Go, а сервер на Python, они поймут друг друга идеально, потому что код сгенерирован из одного контракта.

## 3. Четыре типа стриминга в gRPC
Если REST — это всегда "запрос-ответ", то gRPC благодаря HTTP/2 поддерживает мощные стримы (потоки).

1. **Unary (Одинарный):** Классика. Клиент отправил один запрос, сервер вернул один ответ.
2. **Server Streaming:** Клиент отправляет один запрос, а сервер в ответ открывает поток и пушит туда много сообщений (например, сервер отправляет видео-поток или логи).
3. **Client Streaming:** Клиент открывает поток и долго шлет данные, а сервер ждет окончания и возвращает один ответ (например, клиент загружает огромный файл по частям).
4. **Bidirectional Streaming (Двунаправленный):** Обе стороны могут одновременно отправлять и читать сообщения в любом порядке (чат в реальном времени или онлайн-игра).

## 4. Недостатки gRPC (Что ответить на интервью)

Если gRPC такой классный, почему мы не используем его везде?
На собеседованиях хотят услышать, что вы знаете не только плюсы, но и минусы технологии.

1. **Не читается человеком:** Вы не можете просто открыть вкладку "Network" в браузере и посмотреть ответ, как это было с JSON. Данные бинарные. Нужны специальные утилиты (BloomRPC, Postman с поддержкой gRPC, `grpcurl`).
2. **Плохая поддержка в браузерах:** Браузеры не дают разработчикам полного контроля над HTTP/2-фреймами. Поэтому напрямую вызвать gRPC из JavaScript/React нельзя. Нужна прослойка (gRPC-Web или шлюз, который переводит REST в gRPC, например, *grpc-gateway*).
3. **Строгая связь (Coupling):** И клиент, и сервер должны иметь скомпилированные файлы из одного `.proto` контракта. Если контракт меняется несовместимым образом, нужно обновлять всех клиентов.

## Шпаргалка: REST vs gRPC

| Характеристика | REST | gRPC |
| :--- | :--- | :--- |
| **Формат данных** | JSON / XML (текст) | Protobuf (бинарный) |
| **Протокол** | Обычно HTTP/1.1 | Строго HTTP/2 |
| **Контракт (Схема)** | Опционально (OpenAPI/Swagger) | Строго обязательно (`.proto` файл) |
| **Скорость** | Медленнее (парсинг текста) | Очень быстро (бинарная сериализация) |
| **Стриминг** | Нет (только WebSockets или SSE) | Да (4 вида стриминга из коробки) |
| **Идеальный юзкейс**| Публичное API для фронтенда и сторонних клиентов | Внутреннее общение микросервисов между собой |
