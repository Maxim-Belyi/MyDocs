# Graceful Shutdown

Graceful Shutdown — это процесс корректного завершения работы приложения, при котором оно перестает принимать новые запросы, но дает время на завершение текущих активных операций перед полной остановкой процесса.

---

## 1. Зачем это нужно? (Пример с RabbitMQ и ClickHouse)

Рассмотрим классическую задачу: воркер (worker) читает батч из 10 000 сообщений из очереди RabbitMQ и записывает их в базу данных ClickHouse. Если приложение завершается некорректно (например, при обновлении контейнера в Kubernetes), возможны два негативных сценария без использования Graceful Shutdown:

* **Сценарий А: Автоматический ACK (подтверждение) в RabbitMQ.**
  Сервис настроен на автоматическое подтверждение доставки. RabbitMQ автоматически стирает у себя эти 10 000 сообщений, как только отдает их сервису. Если сервис падает (завершается операционной системой) на середине записи в ClickHouse, мы **безвозвратно теряем сообщения**, которые не успели записаться.

* **Сценарий Б: Ручной ACK (без Graceful Shutdown).**
  Мы отправляем 10 000 сообщений в сервис с ручным подтверждением. Сервис принудительно выключается во время записи и, так как ACK не был отправлен, возвращает эти сообщения обратно в очередь. Но в это время воркер уже успел записать часть сообщений в ClickHouse. При следующем запуске воркер снова прочитает все 10 000 сообщений, и в базе появятся дубликаты — **мы получаем мусорные данные**.

### Как работает Graceful Shutdown в этом сценарии
Чтобы избежать потерь и дубликатов, процесс завершения строится в строгом порядке, обеспечивающем консистентность:
1. **Остановка приема:** мы закрываем консьюмер RabbitMQ. Приложение больше не берет новые сообщения из очереди.
2. **Ожидание:** дожидаемся завершения работы активных горутин воркеров, которые уже обрабатывают прочитанный батч.
3. **Завершение транзакции:** данные успешно записываются в ClickHouse.
4. **Подтверждение в очереди:** отправляем ручной ACK в RabbitMQ, подтверждая успешную обработку.
5. **Безопасный выход:** процесс завершается. Запущенные бизнес-операции корректно завершены.

## 2. Что делать, если горутина зависла?

Возникает резонный вопрос: *если горутина зависла (например, дедлок БД, бесконечный цикл или "замерзшее" сетевое соединение), то Graceful Shutdown не завершится никогда?*

**Именно так.** Если просто ждать завершения всех горутин (например, бесконечно заблокировавшись на `sync.WaitGroup`), приложение "повиснет" в состоянии завершения. В современных средах (например, Kubernetes) оркестратор подождет определенное время (обычно 30 секунд), после чего просто "убьет" контейнер жестким сигналом (SIGKILL).

**Решение: обязательный контекст с таймаутом.**
Чтобы контролировать процесс остановки, в логику выключения **всегда передается таймаут**. 
Если за отведенное время (например, 5-15 секунд) горутины не успели завершить работу, приложение логирует ошибку и принудительно завершается, чтобы не заблокировать деплой новой версии сервиса.

## 3. Практическая реализация в Go

Для реализации Graceful Shutdown программа перехватывает системные сигналы от операционной системы (обычно `SIGINT` и `SIGTERM`), после чего запускает механизм остановки с ограниченным временем.

```go
package main

import (
    "context"
    "fmt"
    "log"
    "os/signal"
    "sync"
    "syscall"
    "time"
)

func main() {
    // 1. Создаем контекст, который отменится при получении сигналов (Go 1.16+)
    // SIGINT - это Ctrl+C в терминале. SIGTERM - сигнал остановки от Docker/K8s
    ctx, stop := signal.NotifyContext(context.Background(), syscall.SIGINT, syscall.SIGTERM)
    defer stop()

    var wg sync.WaitGroup
    
    // Эмуляция работы консьюмера (чтение RabbitMQ -> запись ClickHouse)
    wg.Add(1)
    go func() {
        defer wg.Done()
        fmt.Println("Воркер начал обработку 10000 сообщений...")
        // Эмуляция долгой обработки. Если поставить больше 5 сек, сработает таймаут Shutdown.
        time.Sleep(3 * time.Second) 
        fmt.Println("Воркер успешно завершил транзакцию и отправил ACK.")
    }()

    // 2. Блокируем главный поток, пока контекст не будет отменен сигналом
    <-ctx.Done()
    log.Println("Получен сигнал на завершение. Начинаем Graceful Shutdown...")
    
    // 3. Отключаем перехват сигналов. Если пользователь нажмет Ctrl+C еще раз - программа упадет немедленно
    stop()

    // 4. Создаем НОВЫЙ контекст с таймаутом 5 секунд для самого этапа завершения
    shutdownCtx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()

    // 5. Запускаем ожидание горутин в отдельном потоке
    done := make(chan struct{})
    go func() {
        wg.Wait()
        // Здесь обычно закрывают коннекты: db.Close(), rabbit.Close()
        close(done)
    }()

    // 6. Ждем, что наступит раньше: горутины завершатся (done) или выйдет время (shutdownCtx.Done)
    select {
    case <-done:
        log.Println("Все операции успешно завершены. Сервис остановлен.")
    case <-shutdownCtx.Done():
        // Сработает, если wg.Wait() "зависнет" дольше 5 секунд
        log.Fatalf("Таймаут Graceful Shutdown. Принудительное завершение: %v", shutdownCtx.Err())
    }
}
```

## Итог: Чек-лист по Graceful Shutdown

1. **Сигналы:** всегда перехватывайте `SIGTERM` и `SIGINT`. Попытка перехватить жесткий сигнал `SIGKILL` (kill -9) приведет к ошибке, его перехватить невозможно.
2. **Остановка приема новых задач:** при получении сигнала первым делом прекратите прием данных (вызовите `server.Shutdown()` для HTTP-сервера или закройте канал консьюмера в брокере сообщений).
3. **Ожидание рабочих горутин:** используйте `sync.WaitGroup` или пакет `golang.org/x/sync/errgroup` для отслеживания текущих активных процессов.
4. **Таймаут остановки:** всегда ограничивайте время Shutdown с помощью `context.WithTimeout`, чтобы зависшая горутина не сломала процесс обновления всего сервиса.
5. **Очистка ресурсов:** закрывайте подключения к базам данных (`db.Close()`) только после завершения всех активных горутин, иначе рабочие горутины упадут с ошибкой обрыва соединения БД.
