# Zap

Пакет `zap` (`go.uber.org/zap`) — это высокопроизводительная библиотека для структурированного логирования от компании Uber. Она известна тем, что работает значительно быстрее стандартного `log` или популярного `logrus`, при этом минимизируя аллокации памяти.

---

## Зачем нужен этот пакет?

Современные системы логирования (такие как ELK Stack, Splunk, Datadog) ожидают логи в структурированном виде, обычно в формате JSON. Стандартный пакет `log` выводит простой текст, который сложно парсить. `zap` решает эту проблему:

1. **Структурированность**: логируются не просто строки текста, а события с набором полей (ключ-значение).
2. **Производительность**: оптимизировано под минимальные задержки и сборку мусора (GC).

---

## Основные компоненты: Logger и SugaredLogger

Библиотека предоставляет два разных логгера:
* **Logger**: максимально быстрый логгер, требующий строгой типизации полей. Идеален для критичных участков (hot paths).
* **SugaredLogger**: более медленный (по меркам `zap`), но с более удобным API (похожим на `fmt.Printf`). Подходит для большинства обычных задач.

### Инициализация
Самый простой способ — использовать готовые конфигурации:
* `zap.NewProduction()`: выводит JSON в `stderr`, уровень логирования INFO и выше.
* `zap.NewDevelopment()`: выводит раскрашенный и читаемый текст в `stderr`, уровень логирования DEBUG и выше.

```go
logger, _ := zap.NewProduction()
defer logger.Sync() // Обязательный вызов для сброса буферов перед завершением
```

## Основные методы (Logger)

Методы строго типизированного логгера. Каждое поле передается через специальные функции `zap.String`, `zap.Int`, `zap.Error` и т.д.

### Info, Error, Debug, Fatal
Записывают событие на соответствующем уровне логирования.

```go
func (log *Logger) Info(msg string, fields ...zap.Field)
```

**Пример:**
```go
logger, _ := zap.NewProduction()

url := "http://example.com"
attempt := 3

logger.Info("не удалось загрузить страницу",
    zap.String("url", url),
    zap.Int("attempt", attempt),
    zap.Duration("backoff", time.Second),
)
```
*Результат (в формате JSON):*
`{"level":"info","ts":1716200000.0,"caller":"main.go:10","msg":"не удалось загрузить страницу","url":"http://example.com","attempt":3,"backoff":1}`

### With
Создает копию логгера, к которому навсегда прикреплены определенные поля. Это крайне полезно, когда вы передаете логгер в обработчик HTTP-запроса, чтобы каждый лог внутри этого запроса автоматически содержал, например, `request_id`.

```go
func (log *Logger) With(fields ...zap.Field) *Logger
```

**Пример:**
```go
requestLogger := logger.With(
    zap.String("request_id", "req-12345"),
    zap.String("user_id", "usr-987"),
)

// Все последующие логи будут включать request_id и user_id
requestLogger.Info("Запрос в базу данных")
requestLogger.Error("Произошла ошибка базы")
```

### Полезные типизированные поля (Fields)
Для максимальной скорости `zap` требует указывать тип поля:
* `zap.String(key, val string)`
* `zap.Int(key, val int)`
* `zap.Bool(key, val bool)`
* `zap.Duration(key, val time.Duration)`
* `zap.Error(err error)` — автоматически использует ключ `"error"`.
* `zap.Any(key, val interface{})` — работает чуть медленнее (через рефлексию), но позволяет передать любую структуру или сложный тип.

## Основные методы (SugaredLogger)

Чтобы получить "сладкий" логгер, нужно вызвать метод `Sugar()` у обычного логгера.

```go
sugar := logger.Sugar()
```

### Infow, Errorw, Debugw
Аналоги базовых методов, но поля передаются как пары ключ-значение (interface{}).

```go
sugar.Infow("не удалось загрузить страницу",
    "url", url,
    "attempt", attempt,
    "backoff", time.Second,
)
```

### Infof, Errorf, Debugf
Используются для форматирования строк аналогично `fmt.Printf`.

```go
sugar.Infof("Пользователь %s успешно авторизован, попытка %d", username, attempt)
```

## Итог: Чек-лист по пакету zap

1. **Тип логгера:** для большинства микросервисов и бизнес-логики используйте `SugaredLogger`. Строго типизированный `Logger` используйте только там, где критична каждая наносекунда.
2. **Синхронизация:** всегда вызывайте `defer logger.Sync()` в функции `main`, чтобы гарантировать запись логов из памяти на диск при завершении программы.
3. **Контекст:** используйте структурированные поля (например, `zap.String("user_id", "123")`) вместо внедрения значений прямо в сообщение (`"пользователь 123"`). Это позволит легко искать события по `user_id` в Kibana/Datadog.
