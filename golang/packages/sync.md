# sync

Пакет `sync` предоставляет базовые примитивы синхронизации, такие как мьютексы и группы ожидания. Он необходим для безопасной работы с общими данными из разных горутин и управления их жизненным циклом.

---

## 1. sync.Mutex (Взаимное исключение)

`sync.Mutex` (мьютекс) используется для защиты критических секций кода. Он гарантирует, что только одна горутина может получить доступ к защищенному участку кода или переменной в любой момент времени.

### Основные методы:
*   `Lock()`: Захватывает мьютекс. Если мьютекс уже захвачен другой горутиной, текущая горутина заблокируется и будет ждать его освобождения.
*   `Unlock()`: Освобождает мьютекс. **Внимание:** вызов `Unlock` на незаблокированном мьютексе приведет к панике (`fatal error: sync: unlock of unlocked mutex`). Всегда используйте `defer mutex.Unlock()` сразу после `Lock()`.
*   `TryLock() bool` *(с Go 1.18)*: Пытается захватить мьютекс. Если он свободен, захватывает его и возвращает `true`. Если занят, не блокирует горутину, а сразу возвращает `false`.

**Пример:**
```go
type Counter struct {
    mu    sync.Mutex
    value int
}

func (c *Counter) Increment() {
    c.mu.Lock()
    defer c.mu.Unlock() // Гарантирует освобождение даже при досрочном return или panic
    c.value++
}
```

## 2. sync.RWMutex (Мьютекс для чтения/записи)

`sync.RWMutex` оптимизирован для сценариев, где данные часто читаются, но редко изменяются. Он позволяет **многим горутинам одновременно читать** данные, но **только одной горутине писать** данные (при этом блокируются как все новые читатели, так и другие писатели).

### Основные методы:
*   `RLock()`: Захватывает мьютекс для **чтения**. Если мьютекс захвачен для записи, горутина заблокируется.
*   `RUnlock()`: Освобождает мьютекс после чтения.
*   `Lock()` / `Unlock()`: Захватывает мьютекс для **записи**.
*   `TryRLock() bool` / `TryLock() bool`: Неблокирующие попытки захвата.

**Пример:**
```go
type SafeCache struct {
    mu   sync.RWMutex
    data map[string]string
}

// Читатели не блокируют друг друга, чтение идет параллельно
func (c *SafeCache) Get(key string) (string, bool) {
    c.mu.RLock()
    defer c.mu.RUnlock()
    val, ok := c.data[key]
    return val, ok
}

// Запись блокирует всех
func (c *SafeCache) Set(key, value string) {
    c.mu.Lock()
    defer c.mu.Unlock()
    c.data[key] = value
}
```

## 3. sync.WaitGroup (Группа ожидания)

Используется для ожидания завершения пула (группы) горутин. Это стандартный способ дождаться выполнения асинхронных задач перед выходом из программы или продолжением логики.

### Основные методы:
*   `Add(delta int)`: Увеличивает внутренний счетчик на `delta`. Вызывается **строго до** запуска горутины (в родительском потоке).
*   `Done()`: Уменьшает счетчик на 1 (синтаксический сахар для `Add(-1)`). Обычно вызывается через `defer` внутри самой горутины.
*   `Wait()`: Блокирует текущую горутину, пока счетчик не станет равен 0.

**Пример:**
```go
var wg sync.WaitGroup

for i := 1; i <= 3; i++ {
    wg.Add(1) // Добавляем перед запуском горутины!
    go func(id int) {
        defer wg.Done() // Обязательно вызываем при выходе
        fmt.Printf("Горутина %d завершена\n", id)
    }(i)
}

wg.Wait() // Ждем, пока все 3 горутины не завершатся
```

## 4. sync.Once (Однократное выполнение)

Гарантирует, что определенная функция будет выполнена ровно один раз, независимо от того, сколько горутин попытаются её вызвать одновременно. Часто используется для ленивой инициализации соединений с БД, синглтонов или чтения тяжелых конфигов.

### Основные методы:
*   `Do(f func())`: Выполняет функцию `f`, если `Do` вызывается для этого экземпляра `sync.Once` впервые. При повторных вызовах `Do` функция `f` просто игнорируется.

*(Начиная с Go 1.21 добавлены глобальные функции-хэлперы: `sync.OnceFunc`, `sync.OnceValue` и `sync.OnceValues`, которые оборачивают вашу функцию и позволяют возвращать значение/ошибку).*

**Пример:**
```go
var (
    once sync.Once
    db   *Database
)

func GetDatabase() *Database {
    once.Do(func() {
        fmt.Println("Инициализация БД (вызовется лишь однажды)...")
        db = &Database{}
    })
    return db
}
```

## 5. sync.Map (Потокобезопасная Map)

Стандартная мапа (`map[K]V`) не является потокобезопасной — конкурентная запись в нее вызовет панику `fatal error: concurrent map writes`. `sync.Map` решает эту проблему без явного использования мьютексов.

> **Важно:** `sync.Map` часто медленнее обычной мапы, обернутой в `sync.RWMutex`. В официальной документации сказано, что `sync.Map` рекомендуется использовать только в двух случаях:
> 1. Ключи стабильны (записываются один раз) и читаются **очень** часто из разных горутин (например, кэши).
> 2. Разные горутины работают с совершенно разными непересекающимися наборами ключей.

### Основные методы (значения имеют тип `any`):
*   `Store(key, value any)`: Записывает значение по ключу.
*   `Load(key any) (value any, ok bool)`: Читает значение.
*   `LoadOrStore(key, value any) (actual any, loaded bool)`: Читает значение. Если его нет — записывает переданное. Возвращает `loaded = true`, если значение уже существовало.
*   `Delete(key any)`: Удаляет значение.
*   `LoadAndDelete(key any) (value any, loaded bool)`: Читает и сразу атомарно удаляет ключ.
*   `CompareAndSwap(key, old, new any) bool`: (Атомарно) записывает `new`, только если текущее значение равно `old`.
*   `CompareAndDelete(key, old any) bool`: (Атомарно) удаляет ключ, только если его значение равно `old`.
*   `Swap(key, value any) (previous any, loaded bool)`: Безусловно записывает новое значение и возвращает старое.
*   `Range(f func(key, value any) bool)`: Перебирает все элементы. Если функция `f` возвращает `false`, перебор останавливается.

**Пример:**
```go
var sm sync.Map

// Запись
sm.Store("host", "localhost")

// Чтение (требуется приведение типов - Type Assertion)
if val, ok := sm.Load("host"); ok {
    fmt.Println(val.(string)) 
}

// Перебор
sm.Range(func(key, value any) bool {
    fmt.Printf("%v: %v\n", key, value)
    return true // Продолжаем перебор
})
```

## 6. sync.Pool (Пул объектов)

Кэширует выделенные, но больше не используемые объекты для их повторного использования. Это значительно снижает нагрузку на сборщик мусора (Garbage Collector, GC) при частых аллокациях одинаковых структур (например, буферов `bytes.Buffer` в HTTP-сервере).

### Основные поля и методы:
*   `New func() any`: Поле конфигурации структуры. Определяет функцию создания нового объекта, если пул пуст.
*   `Get() any`: Достает объект из пула. Если пул пуст, вызывает `New`. Возвращает `any`, требует приведения типа.
*   `Put(x any)`: Возвращает объект в пул.

> **Критически важно:** перед возвратом объекта в пул (или сразу после изъятия) его **нужно очистить (сбросить состояние)**, иначе следующая горутина получит объект с чужими (старыми) данными. Сборщик мусора может в любой момент очистить `sync.Pool`, поэтому полагаться на то, что объект там сохранится навсегда, нельзя.

**Пример:**
```go
import (
    "bytes"
    "sync"
)

var bufPool = sync.Pool{
    New: func() any {
        return new(bytes.Buffer)
    },
}

func logData(data string) {
    // Берем буфер из пула (быстро)
    buf := bufPool.Get().(*bytes.Buffer)
    
    // ОЧИЩАЕМ его от старых данных перед использованием!
    buf.Reset() 
    
    buf.WriteString(data)
    
    // Возвращаем буфер обратно в пул для повторного использования
    bufPool.Put(buf)
}
```

## 7. sync.Cond (Условная переменная)

Используется для блокировки горутин до тех пор, пока не наступит определенное условие. Это мощный, но низкоуровневый инструмент, который на практике чаще всего заменяют каналами (channels), так как они безопаснее и проще.

### Основные методы:
*   `NewCond(l Locker) *Cond`: Создает `Cond`, привязанный к переданному мьютексу `l`.
*   `Wait()`: Атомарно разблокирует привязанный мьютекс и приостанавливает выполнение горутины. При пробуждении (через `Signal` или `Broadcast`) заново захватывает мьютекс перед продолжением.
*   `Signal()`: Будит **одну** случайную горутину, ожидающую в `Wait()`.
*   `Broadcast()`: Будит **все** горутины, ожидающие в `Wait()`.

## Итог: Чек-лист по пакету sync

1. **Копирование запрещено:** Структуры из пакета `sync` (`Mutex`, `WaitGroup`, `Pool` и т.д.) **нельзя копировать** по значению. Если вы передаете их в функции, передавайте указатель `*sync.Mutex` или используйте в составе структур, передаваемых по ссылке. При копировании ломается внутреннее состояние блокировок.
2. **Дедлоки (Deadlocks):** При использовании `Mutex` и `RWMutex` всегда используйте `defer mu.Unlock()` сразу после `mu.Lock()`.
3. **wg.Add():** Вызывайте `Add` только **снаружи**, до ключевого слова `go`. Если вызвать `Add` внутри самой горутины, `Wait` в главном потоке может сработать раньше, чем горутина успеет вызваться и сделать `Add`.
4. **sync.Map:** В 95% случаев обычная `map + sync.RWMutex` работает быстрее и предоставляет строгую типизацию. `sync.Map` нужна только для очень специфичных высоконагруженных кэшей.
5. **sync.Pool:** Обязательно очищайте состояние объекта через метод `Reset()` или вручную перед возвратом в пул.
