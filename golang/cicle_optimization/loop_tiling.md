# Тайлинг (Loop Tiling) & Loop Interchange (Перестановка циклов)

При работе с многомерными массивами (матрицами) производительность часто упирается не в скорость процессора, а в скорость оперативной памяти. Когда мы читаем двумерный массив "по колонкам", мы постоянно перепрыгиваем огромные участки памяти. Процессор не может кэшировать такие скачки, и возникает множество промахов кэша (Cache Misses).

Рассмотрим классическую задачу умножения матриц ($O(N^3)$) размером 1024x1024 (по 8 МБ каждая) и попытаемся ее оптимизировать.

Все бенчмарки запущены на **AMD Ryzen 5 7500F**.

---

## Loop Tiling (Блочная оптимизация)

В классической Computer Science (и в языках вроде C++) для решения проблемы кэша применяют **Loop Tiling (Тайлинг)**. Огромная матрица виртуально разбивается на маленькие блоки (например, 64x64 элемента). Этот квадратик идеально помещается в самый быстрый кэш L1 (32 КБ) и обрабатывается без обращений к медленной RAM.

Сравним классический алгоритм в лоб и сложный алгоритм с Тайлингом (6 вложенных циклов):

*   Классический цикл (Standard): `3,080,132,512 ns/op` (~3.08 сек)
*   Блочный цикл (Tiled): `2,904,941,966 ns/op` (~2.90 сек)

**Ускорение:** ~5.8%.
**Почему так?** 

1.  **Аппаратные предвыборщики (Hardware Prefetchers):** Современные процессоры невероятно умны. Предвыборщик Ryzen заметил, что мы прыгаем по памяти с фиксированным шагом (по 1024 элемента), и начал подтягивать нужные строки в кэш *заранее*, аппаратно сгладив нашу плохую работу с памятью.
2.  **Проклятие Bounds Checking в Go:** Компилятор Go вставляет невидимые проверки выхода за границы массива (`if i >= len { panic }`). В тайлинге у нас 6 вложенных циклов со сложной арифметикой индексов. Компилятор не смог доказать их безопасность и оставил проверки внутри самого глубокого цикла. Накладные расходы на них съели всю выгоду от кэша!

## Попытка №2: Loop Interchange (Перестановка циклов)

Если сложные блокировки не работают, обратимся к "Механической симпатии" (Mechanical Sympathy) — пониманию того, как "железо" любит получать данные. 

Любой процессор обожает **строго линейный доступ к памяти**. Что если просто поменять местами вложенные циклы `i-j-k` на `i-k-j`?

```go
func BenchmarkMatrixMultInterchange(b *testing.B) {
	A, B, C := make([]int, N*N), make([]int, N*N), make([]int, N*N)

	b.ResetTimer()
	for n := 0; n < b.N; n++ {
		// Порядок изменен: i -> k -> j
		for i := 0; i < N; i++ {
			for k := 0; k < N; k++ {
				a := A[i*N+k] // Вычисляется 1 раз и ложится в регистр
				
				// Внутренний цикл идет по 'j'
				for j := 0; j < N; j++ {
					// Идеально линейный доступ: [j], [j+1], [j+2]
					C[i*N+j] += a * B[k*N+j]
				}
			}
		}
	}
}
```

*   Классический цикл: `3,080,132,512 ns/op` (~3.08 сек)
*   **Loop Interchange:** `551,534,306 ns/op` (~0.55 сек)

**Ускорение:** В **5.5 РАЗ**

### Чем этот подхож лучше?
1. В самом внутреннем цикле переменная `j` увеличивается на 1.
2. Массив `C[i*N+j]` читается и пишется строго последовательно (0, 1, 2, 3...).
3. Массив `B[k*N+j]` читается строго последовательно!
4. Никаких прыжков по памяти. Hardware Prefetcher работает в идеальных тепличных условиях. Кроме того, компилятору Go намного проще оптимизировать проверки границ для линейно растущего индекса `j`.

## Выводы 

1.  **Линейный доступ рулит:** Независимо от процессора (Intel, AMD или ARM), последовательное чтение памяти всегда будет в разы быстрее хаотичного. Проектируйте алгоритмы так, чтобы внутренние циклы шли по массиву строго вперед `[0], [1], [2]`.
2.  **Go — не C++:** Оптимизации, пришедшие из старых учебников для C/Fortran (вроде ручного Тайлинга), в Go часто спотыкаются о сборщик мусора и встроенные проверки границ массивов (Bounds Checking).
3.  **Не усложняйте без тестов:** Элегантное решение с перестановкой 3 строчек кода оказалось в десятки раз эффективнее и читаемее, чем сложный алгоритм с 6 циклами. Всегда проверяйте свои гипотезы через `go test -bench`.
