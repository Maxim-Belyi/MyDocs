Оптимизация Loop Unrolling

Компилятор Go достаточно умен, чтобы самостоятельно оптимизировать многие базовые вещи, но когда дело доходит до миллисекунд в высоконагруженных системах (hot paths), приходится применять микро-оптимизации, учитывающие архитектуру процессора.

(CPU: AMD Ryzen 5 7500F).

Базовый стенд (Контрольная точка)​

Сравним классический цикл заполнения слайса (размером 1024 элемента)

const size = 1 << 10 // 1024

func BenchmarkWithoutOptimization(b *testing.B) {
	data := make([]int, size)
	for i := 0; i < b.N; i++ {
		for j := 0; j < len(data); j++ {
			data[j] = i
		}
	}
}

Результат: 421.9 ns/op (наносекунд на операцию)

Вычисление длины (len)​

Кешируем длинну массива

func BenchmarkWithCalculatedLenght(b *testing.B) {
	data := make([]int, size)
	for i := 0; i < b.N; i++ {
		lenght := len(data) // Кешируем длину
		for j := 0; j < lenght; j++ {
			data[j] = i
		}
	}
}

Результат: 441.8 ns/op (Даже чуть медленнее базового, в пределах погрешности).

Почему это не работает? В Go len() — это не функция. Слайс под капотом — это структура, где длина (len) лежит как готовое поле (число int). Компилятор заменяет len(data) на простое чтение переменной. Кеширование здесь бесполезно и только добавляет лишнюю переменную в регистр процессора.

Loop Unrolling (Раскрутка цикла)​

Loop Unrolling — это искусственное увеличение количества полезных операций внутри одной итерации с уменьшением количества самих итераций.

Пробуем шаг х8 (обрабатываем по 8 элементов за раз):

func BenchmarkWithLoopUnwinding(b *testing.B) {
	data := make([]int, size)
	for i := 0; i < b.N; i++ {
		// Шаг цикла теперь 8, а не 1
		for j := 0; j < len(data); j+=8 {
			data[j] = i;   data[j+1] = i; data[j+2] = i; data[j+3] = i
			data[j+4] = i; data[j+5] = i; data[j+6] = i; data[j+7] = i
		}
	}
}

Результат (х8): 54.10 ns/op (Ускорение почти в 8 РАЗ). Результат (х16): 32.23 ns/op (Ускорение в 13 РАЗ).

За счет чего такой колоссальный прирост?​

  1. Уменьшение Overhead'а: Мы в 8/16 раз реже делаем инкремент (j += 8), реже сравниваем с длиной (j < len) и реже делаем прыжок в начало цикла (Jump).
  2. Instruction-Level Parallelism (ILP): Современные CPU суперскалярны. Они могут выполнять несколько независимых инструкций в один такт часов. Процессор видит data[j+1] = i и data[j+2] = i, понимает, что они не зависят друг от друга, и выполняет их параллельно на разных исполнительных блоках ядра.
  3. Branch Prediction (Предсказатель ветвлений): Чем меньше переходов (if / условия цикла), тем реже процессор ошибается и очищает конвейер (pipeline flush).

Экспоненциальное копирование​

Что делать, если мы не хотим писать ужасный, нечитаемый код с 16 строками присваивания, но хотим скорость? Встроенная функция copy() в Go реализована на низкоуровневом ассемблере (runtime.memmove) и использует векторные инструкции процессора (AVX/SIMD).

Можно использовать алгоритм "логарифмического роста":

  1. Записываем 1 элемент.
  2. Копируем 1 элемент (получаем 2).
  3. Копируем 2 элемента (получаем 4).
  4. Копируем 4 (получаем 8) и так далее.
func BenchmarkWithBuiltinCopy(b *testing.B) {
    data := make([]int, size)
    for i := 0; i < b.N; i++ {
        data[0] = i
        // Копируем уже заполненные элементы в незаполненную часть
        for j := 1; j < len(data); j *= 2 {
            copy(data[j:], data[:j])
        }
    }
}

Результат: 76.32 ns/op (В 5.5 раз быстрее базы!).

Этот метод немного проигрывает жесткому x16 Unrolling'у на небольших массивах (1024), но часто обгоняет его на гигантских слайсах, так как runtime.memmove максимально оптимизирован для переноса больших блоков памяти. К тому же, этот код гораздо компактнее.

Замечание про Bounds Check Elimination (BCE) и SIMD​

Можно попробовать помочь компилятору созданием tmp := data[:size].

func BenchmarkWithAutoSIMD(b *testing.B) {
	data := make([]int, size)
	for i := 0; i < b.N; i++ {
		tmp := data[:size] // Попытка подсказать компилятору
		for j := 0; j < len(tmp); j++ {
			tmp[j] = i
		}
	}
}

Результат: 442.0 ns/op (Как у базового).

Почему нет ускорения? BCE (Bounds Check Elimination) — это процесс, когда компилятор убирает скрытые проверки if j >= len(data) { panic } внутри цикла. В нашем случае, базовый цикл for j := 0; j < len(data); j++ уже идеально понятен компилятору Go. Он сам видит, что j никогда не превысит длину массива, и автоматически удаляет проверки границ. Поэтому дополнительная подсказка tmp := data[:size] ничего не меняет. Компилятор уже выжал из простого цикла всё, что мог, без ручного развертывания (Unrolling).

Выводы​

  1. Не кешируйте len() слайсов и строк — компилятор умнее нас, это не даст прироста (а иногда даже замедлит код).
  2. Loop Unrolling (x4 / x8 / x16) дает гигантский буст производительности на математических операциях и присвоениях за счет архитектуры процессора (ILP).
  3. Используйте copy() с экспоненциальным ростом для быстрого заполнения больших слайсов одним значением — это компромисс между безумной скоростью и читаемостью кода.
  4. Главное правило: Никогда не применяйте эти хаки «вслепую». Делайте это только после профилирования (pprof), когда вы уверены, что именно этот цикл тормозит приложение. Развертывание цикла ухудшает читаемость кода, поэтому оно должно быть оправдано.
