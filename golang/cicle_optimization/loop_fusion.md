# Loop Fusion (Слияние) и CPU Cache

Современные процессоры работают в сотни раз быстрее, чем оперативная память. Чтобы не простаивать, процессор копирует данные из RAM в свой сверхбыстрый, но очень маленький кэш (L1/L2/L3). 

**Суть Loop Fusion:** Если нам нужно сделать несколько разных проходов по одному и тому же массиву данных, мы объединяем их в один цикл. Это позволяет процессору загрузить кусок данных в быстрый кэш L1 ровно *один раз*, выполнить с ним все необходимые операции и забыть, не читая его из медленной памяти повторно.

(Размер массива: 1 миллион элементов = ~8 МБ памяти. CPU: AMD Ryzen 5 7500F).

---

## Сценарий 1: Преобразование данных (Мутация)

Представим пайплайн обработки данных: мы берем массив `A`, умножаем его на 2 и кладем в массив `B`. Затем берем массив `B`, прибавляем к нему 5 и кладем в `C`.

**Раздельные циклы:**
```go
// BenchmarkTwoLoops
for j := 0; j < len(dataA); j++ {
    dataB[j] = dataA[j] * 2
}
for j := 0; j < len(dataB); j++ {
    dataC[j] = dataB[j] + 5
}
```

**Слитый цикл (Fused):**
```go
// BenchmarkFusedLoop
for j := 0; j < len(dataA); j++ {
    dataB[j] = dataA[j] * 2
    dataC[j] = dataB[j] + 5
}
```

### Результаты бенчмарка:
*   Два цикла: `777,115 ns/op`
*   Слитый цикл: `517,344 ns/op`
*   **Ускорение: ~33% (в 1.5 раза быстрее)**

**Почему это произошло?**
В первом варианте процессор записал массив `B` (8 Мегабайт) в память. Размер кэша L1/L2 процессора обычно меньше 8 МБ. Это значит, что пока первый цикл дошел до конца массива, начало массива `B` уже было **вытеснено (evicted)** из быстрого кэша обратно в медленную RAM. 
Когда начался второй цикл, процессору пришлось *заново* запрашивать данные `dataB` из оперативной памяти! 
Во втором варианте мы читаем `A`, вычисляем `B` (он всё еще лежит прямо в регистрах процессора или в L1!) и мгновенно вычисляем `C`. Промахи кэша (Cache Misses) сведены к нулю.

## Сценарий 2: Агрегация данных (Чтение)

Нам нужно посчитать сумму всех элементов и одновременно найти максимальное значение. Многие пишут это разными циклами (например, используя готовые функции `Sum(arr)` и `Max(arr)`).

**Раздельные циклы:**
```go
// BenchmarkSeparateAggregation
for j := 0; j < len(data); j++ {
    sum += data[j]
}
for j := 0; j < len(data); j++ {
    if data[j] > max {
        max = data[j]
    }
}
```

**Слитый цикл (Fused):**
```go
// BenchmarkFusedAggregation
for j := 0; j < len(data); j++ {
    sum += data[j]
    if data[j] > max {
        max = data[j]
    }
}
```

### Результаты бенчмарка:
*   Два цикла: `646,166 ns/op`
*   Слитый цикл: `346,379 ns/op`
*   **Ускорение: ~46% (почти в 1.9 раза быстрее)**

**Секрет успеха:**
Помимо очевидного выигрыша от кэша (мы читаем `data[j]` из медленной RAM только один раз вместо двух), мы получаем мощный бонус от снижения **Loop Overhead (накладных расходов цикла)**.
В варианте с двумя циклами мы в два раза больше делаем инкремент счетчика `j++`, в два раза чаще проверяем выход за границы массива `j < len(data)` и делаем в два раза больше прыжков (Jumps) в начало цикла.

##  Выводы

1.  **Берегите кэш процессора:** Чтение из L1 кэша занимает ~1 наносекунду. Чтение из RAM — ~100 наносекунд. Если вы проходитесь по огромному массиву данных несколько раз, вы заставляете процессор ждать оперативную память. Сливайте циклы (Loop Fusion)!
2.  **Закон "Единой ответственности" имеет цену:** В Clean Code нас учат, что функция должна делать что-то одно (только считать сумму или только искать максимум). Но в high-load участках кода архитектурная чистота вступает в конфликт с производительностью. Оптимизированный код часто выглядит менее "красивым".
3.  **Комбинируйте с Loop Unrolling:** Если вы сольете циклы вместе (Fusion), а затем раскрутите полученный цикл (Unrolling x4/x8), вы получите максимально возможную производительность.
4.  **Когда НЕ нужно сливать?** Если внутри первого цикла вы фильтруете элементы (уменьшаете массив в 100 раз), а во втором делаете тяжелую математику над оставшимися — лучше оставить их раздельными.
