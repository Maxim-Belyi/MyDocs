# Оптимизации: Strength Reduction (Снижение стоимости)

**Strength Reduction (Снижение стоимости операций)** — это техника замены "дорогих" математических инструкций для процессора на более "дешевые", дающие точно такой же результат.

В мире процессоров не вся математика равна:
*   `+`, `-`, `&`, `|`, `<<`, `>>` (Сложение, побитовые операции) — выполняются молниеносно, за 1 такт.
*   `*` (Умножение) — чуть медленнее, но аппаратно оптимизировано (1-3 такта).
*   `/`, `%` (Деление и остаток) — самые тяжелые и медленные инструкции процессора (до 15-40 тактов).

Современные компиляторы умеют делать Strength Reduction автоматически. Но, как показывают бенчмарки компилятор спасает нас далеко не всегда.

Все бенчмарки запущены на **AMD Ryzen 5 7500F**.

## Сценарий 1: Остаток от деления (`%`)

Очень частая задача: сделать что-то с каждым 8-м элементом, или распределить запросы по серверам. Для этого используют остаток от деления (Modulo). 

Если делитель — это **степень двойки** (2, 4, 8, 16...), операцию `x % 8` можно математически заменить на побитовое И: `x & 7` (число на единицу меньше).

**Код:**
```go
// Дорого: Остаток от деления
if data[j] % 8 == 0 { ... }

// Дешево: Побитовое "И" (Bitwise AND)
if data[j] & 7 == 0 { ... }
```

### Результаты бенчмарка:
*   Использование `% 8`: `417,988 ns/op`
*   Использование `& 7`: `213,849 ns/op`
*   **Ускорение: Почти в 2 РАЗА (~95% прироста)**

**Почему компилятор Go не сделал это сам?**
Главная проблема в типах данных. Тип `int` в Go — это **знаковое** число (может быть отрицательным). Математика остатка от деления для отрицательных чисел (например, `-5 % 8`) работает иначе, чем побитовое `&`. 
Компилятор *не знает*, какие числа лежат в вашем массиве. Чтобы код гарантированно работал правильно для любых чисел, компилятор вынужден генерировать сложную и медленную последовательность инструкций. 
Но если вы (как разработчик) точно знаете, что индексы/значения положительные — использование `& 7` даст отличный прирост

## Сценарий 2: Умножение

В старых учебниках по C/C++ писали: *"Умножение индекса в цикле — это дорого. Заведите отдельную переменную и складывайте её"*.

Проверим этот миф:
**Код:**
```go
// "Дорогое" умножение на каждой итерации
for j := 0; j < len(data)/8; j++ {
    data[j*8] = j 
}

// "Дешевое" сложение
idx := 0
for j := 0; j < len(data)/8; j++ {
    data[idx] = j
    idx += 8 
}
```

### Результаты бенчмарка:
*   Использование `j * 8`: `64,637 ns/op`
*   Использование `idx += 8`: `66,735 ns/op`
*   **Вердикт: Ручная оптимизация сделала код МЕДЛЕННЕЕ**

**Почему так:**
Здесь мы столкнулись с эволюцией компиляторов и процессоров.
1.  **Умножение на константу "бесплатно":** Компилятор Go видит `j * 8` (8 — это $2^3$) и на этапе компиляции автоматически заменяет умножение на молниеносный побитовый сдвиг `j << 3`. Это и есть автоматический Strength Reduction
2.  **Ловушка зависимости данных (Data Dependency):** Введя переменную `idx += 8`, мы создали зависимость. Следующая итерация цикла *не может* вычислить свой `idx`, пока предыдущая итерация не закончит сложение. Это ломает способность современного процессора выполнять несколько инструкций параллельно (ILP). А вот `j * 8` можно вычислить независимо для любой итерации.

## Выводы 

1.  **Побитовые маски вместо `%`:** Если вам нужно найти остаток от деления на степени двойки (состояние буфера, шардинг, кольцевые очереди), всегда используйте `& (N-1)`. Эта микро-оптимизация стабильно ускоряет код в 2 раза на любом "железе".
2.  *(Пример из реального мира)*: Посмотрите исходный код пакета `sync.Pool` в Go. Вы увидите там именно побитовые маски для работы с очередями, а не знак процента
3.  **Не трогайте умножение:** Оставьте базовую математику компилятору. Ручная замена умножения на сложение с введением новых переменных убивает ILP (параллелизм уровня инструкций) и лишь замедляет код. 
4.  **Учитывайте типы:** Если вы используете тип `uint` (беззнаковое число), компилятор Go ведет себя намного агрессивнее и часто сам заменяет `% 8` на `& 7`. Строгая типизация помогает компилятору оптимизировать ваш код.
