# Software Pipelining 

**Software Pipelining (Программный конвейер)** — это продвинутая техника перераспределения кода. Ее суть: пока процессор выполняет математику для *текущей* итерации цикла, мы заставляем его параллельно загружать из памяти данные для *следующей* итерации, чтобы скрыть медленную задержку оперативной памяти (Memory Stall).

В языках вроде C++ при написании высокопроизводительных движков эта техника часто применяется вручную (или флагами компилятора). 
Все бенчмарки запущены на **AMD Ryzen 5 7500F**.

```go
package main

import (
	"testing"
)

// go test -bench=. pipeline_test.go

const size = 1 << 20

// СЦЕНАРИЙ 1: Стандартный цикл (Последовательный)
func BenchmarkStandardLoop(b *testing.B) {
	A := make([]int, size)
	B := make([]int, size)
	C := make([]int, size)
	for k := 0; k < size; k++ {
		B[k], C[k] = k, k+1
	}

	b.ResetTimer()
	for i := 0; i < b.N; i++ {
		// Шаги: Load(B,C) -> Compute -> Store(A)
		for j := 0; j < len(A); j++ {
			A[j] = B[j] * C[j] + B[j]
		}
	}
}

// СЦЕНАРИЙ 2: Interleaved (Чередование инструкций)
// Это комбинация Unrolling + ручного планирования конвейера
func BenchmarkInterleavedPipelining(b *testing.B) {
	A := make([]int, size)
	B := make([]int, size)
	C := make([]int, size)
	for k := 0; k < size; k++ {
		B[k], C[k] = k, k+1
	}

	b.ResetTimer()
	for i := 0; i < b.N; i++ {
		for j := 0; j < len(A); j += 4 {
			// ЭТАП 1: ЗАГРУЗКА (Load)
			// Заставляем CPU параллельно грузить данные сразу для 4 итераций
			b0, c0 := B[j], C[j]
			b1, c1 := B[j+1], C[j+1]
			b2, c2 := B[j+2], C[j+2]
			b3, c3 := B[j+3], C[j+3]

			// ЭТАП 2: МАТЕМАТИКА (Compute)
			// Пока одни данные еще грузятся в кэш, другие уже перемножаются
			res0 := b0*c0 + b0
			res1 := b1*c1 + b1
			res2 := b2*c2 + b2
			res3 := b3*c3 + b3

			// ЭТАП 3: СОХРАНЕНИЕ (Store)
			A[j] = res0
			A[j+1] = res1
			A[j+2] = res2
			A[j+3] = res3
		}
	}
}

// СЦЕНАРИЙ 3: Истинный Software Pipelining (Нахлест итераций)
// Делаем математику для текущего шага, а загрузку данных - для СЛЕДУЮЩЕГО
func BenchmarkTrueSoftwarePipelining(b *testing.B) {
	A := make([]int, size)
	B := make([]int, size)
	C := make([]int, size)
	for k := 0; k < size; k++ {
		B[k], C[k] = k, k+1
	}

	b.ResetTimer()
	for i := 0; i < b.N; i++ {
		// Пролог конвейера: Загружаем данные для САМОЙ ПЕРВОЙ итерации
		b_next, c_next := B[0], C[0]

		// Основной цикл
		for j := 0; j < len(A)-1; j++ {
			// Берем данные, загруженные на предыдущем шаге
			b_curr, c_curr := b_next, c_next
			
			// КОНВЕЙЕР: Грузим данные для СЛЕДУЮЩЕГО шага (скрываем задержку памяти)
			b_next, c_next = B[j+1], C[j+1]

			// Математика для ТЕКУЩЕГО шага
			A[j] = b_curr*c_curr + b_curr
		}

		// Эпилог конвейера: Досчитываем последнюю итерацию
		A[len(A)-1] = b_next*c_next + b_next
	}
}
```

### Результаты бенчмарков:
| Сценарий | Описание | Время выполнения | Результат |
| :--- | :--- | :--- | :--- |
| **Standard Loop** | Обычный цикл | `470,653 ns/op` | База |
| **Interleaved** | Группировка: Load -> Compute -> Store | `427,311 ns/op` | **Ускорение ~9%** |
| **True Pipelining** | Загрузка данных для шага `i+1` на шаге `i` | `481,731 ns/op` | **Замедление ~2.3%** |

---

### 1. Магия Out-of-Order Execution (OoOE)
Современные процессоры (и архитектура Zen 4 в частности) имеют встроенный аппаратный механизм **Внеочередного выполнения (Out-of-Order Execution)**. 
Процессор не выполняет ваш код строчка за строчкой, он аппаратно заглядывает на десятки инструкций вперед, находит те, которые не зависят от текущих вычислений, и выполняет их в фоновом режиме. 
Пока базовая версия (`Standard Loop`) считала математику, Ryzen *сам* догадался загрузить данные для следующих итераций, попытки сделать это вручную лишь помешали аппаратному предсказателю.

### 2. Лишние регистры и переменные
Написав `b_curr = b_next` и `b_next = B[j+1]`, мы заставили компилятор генерировать дополнительные инструкции перемещения данных между регистрами (MOV). Процессор потратил драгоценные такты на перекладывание значений из одной переменной в другую.

### 3. Скрытые Bounds Checks
Компилятору Go намного сложнее доказать безопасность конструкции с индексами `[j+1]`. Вполне вероятно, что он не смог вырезать проверку выхода за границы массива (BCE), и на каждой итерации происходил скрытый `if`, съедающий процессорное время.

---

## Почему Interleaved (Чередование) сработало?

Второй сценарий (`Interleaved`) показал **ускорение на 9%**. 
Здесь было скомбинировано *Loop Unrolling* (шаг х4) и ручное чередование: сначала 4 чтения, потом 4 вычисления, потом 4 записи.

Это максимизирует **ILP (Instruction-Level Parallelism)**. Мы дали процессору 4 абсолютно независимые команды на чтение памяти подряд. Он отправил их в контроллер памяти одним широким пакетом. Затем мы дали 4 независимые математические команды, и он выполнил их на 4 разных ALU-блоках ядра одновременно.

## Выводы 

1.  **Не пишите Software Pipelining вручную в Go:** Оставьте предзагрузку данных аппаратному обеспечению (Hardware Prefetchers) и внеочередному выполнению (OoOE). Умный код в Go часто работает медленнее "тупого", потому что компилятору сложнее его оптимизировать.
2.  **Группировка инструкций (Interleaving):** Если вы уже применяете *Loop Unrolling* (раскрутку), попробуйте сгруппировать все чтения переменных в начале блока, а вычисления — в середине. Это разорвет зависимости между строками кода и позволит процессору выполнять их параллельно.
3.  **Читаемость важнее:** Оптимизация дала 9% прироста ценой превращения 1 строчки понятного кода в 15 строк сложной логики. Применяйте это *только* в криптографии, кодировании видео или в самых горячих путях (Hot Paths) high-load систем. В 99% случаев обычный `for` — это лучший выбор.
