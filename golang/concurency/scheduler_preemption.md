# Планировщик: Кооперативная многозадачность и Вытеснение (Preemption)

Планировщик Go отвечает за распределение десятков тысяч горутин (`G`) по логическим процессорам (`P`) и физическим потокам операционной системы (`M`). В отличие от планировщика ОС (Windows/Linux), который может прервать любой поток в любой момент аппаратно (через таймеры процессора), планировщик Go работает в пользовательском пространстве (User Space).


---

## 1. Кооперативная многозадачность (до Go 1.14)

До выхода версии Go 1.14 планировщик работал по модели **кооперативной многозадачности** с элементами неявного вытеснения.

### Что значит "Кооперативная"?
Это означает, что горутина не прерывается снаружи жестко. Она должна **сама отдать** процессорное время (`yield`), чтобы другие горутины в очереди `P` могли выполниться.

Рантайм Go неявно вставлял проверки в код программы, создавая "точки перехвата" (Safe Points). Горутина отдавала процессор в следующих случаях:
1. **Пролог функции (Function Prologue):** При вызове любой функции компилятор Go вставляет невидимый код, который проверяет размер стека горутины. В этот же момент рантайм мог проверить специальный флаг `stackguard0`. Если планировщик решал, что горутина работает слишком долго (>10 мс), он выставлял этот флаг, и при следующем вызове функции горутина перехватывалась планировщиком (ставилась в конец очереди).
2. **Системные вызовы (Syscalls) и Сеть:** Ожидание ответа по сети, чтение файлов.
3. **Блокировки:** Попытка захватить заблокированный `sync.Mutex` или ожидание `sync.WaitGroup`.
4. **Работа с каналами:** Отправка/получение данных из заблокированного канала.
5. **Явный вызов:** Вызов функции `runtime.Gosched()` программистом вручную.

### Проблема тесных циклов (Tight Loops)
У кооперативной модели была одна фундаментальная уязвимость. Что, если горутина войдет в бесконечный цикл `for`, внутри которого происходят только математические вычисления без вызова других функций, без аллокаций памяти и без системных вызовов?

```go
func main() {
    runtime.GOMAXPROCS(1) // Ограничиваемся 1 ядром для наглядности

    // Запускаем бесконечный "тесный цикл" (tight loop)
    go func() {
        for {
            // Математика без вызовов функций. Точек перехвата (Safe points) нет!
            _ = 1 + 1 
        }
    }()

    time.Sleep(time.Second) // Даем горутине время запуститься
    fmt.Println("Мы никогда не увидим это сообщение в Go 1.13")
}
```

**До Go 1.14:** Горутина захватывала поток ОС (`M`) навсегда. Так как внутри цикла не было вызовов функций (прологов), планировщик просто не получал шанса "вклиниться". Остальные горутины стояли в очереди бесконечно (Starvation). Более того, мог зависнуть сборщик мусора (GC), так как для фазы Stop The World он обязан дождаться, пока все горутины дойдут до Safe Point.

## 2. Асинхронная вытесняющая многозадачность (Go 1.14+)

Для решения проблемы тесных циклов в релизе Go 1.14 была внедрена **Асинхронная вытесняющая многозадачность (Asynchronous Preemption)** на основе системных сигналов.

### Как это работает теперь:

1. **Фоновый мониторинг (sysmon):**
   В рантайме Go всегда работает специальный скрытый системный поток ОС, не привязанный к логическим процессорам (`P`). Он называется **`sysmon`** (System Monitor).
   
2. **Таймаут 10 мс:**
   Поток `sysmon` периодически (примерно каждые 10 мс) сканирует все запущенные горутины. Если он замечает, что какая-то горутина работает на ядре процессора непрерывно дольше 10 миллисекунд, он принимает решение прервать её.

3. **Сигнал SIGURG:**
   Поскольку горутина в тесном цикле не делает вызовов функций, рантайм больше не ждет её на `Safe Point`. Вместо этого `sysmon` отправляет аппаратному потоку ОС (`M`), выполняющему эту горутину, UNIX-сигнал **`SIGURG`** (Urgent Condition on Socket).

4. **Аппаратное прерывание:**
   Поток ОС, получив сигнал `SIGURG`, немедленно прерывает выполнение пользовательского кода на аппаратном уровне (Hardware Interrupt) и вызывает обработчик сигнала, зарегистрированный рантаймом Go (`sigtramp`).

5. **Сохранение контекста и парковка:**
   Обработчик сигнала в рантайме внедряет в стек выполняющейся горутины вызов специальной функции `asyncPreempt`. Как только горутина "просыпается" после сигнала, она невольно вызывает эту функцию, которая сохраняет состояние регистров, ставит горутину обратно в глобальную очередь (Global Run Queue) и освобождает ядро для других ожидающих горутин.

### Исключения и ограничения
Даже с асинхронным вытеснением остаются моменты, когда горутину вытеснить нельзя (например, если она взаимодействует с кодом на языке C через `CGO` и в данный момент выполняет C-код, либо находится в критической секции рантайма со снятым флагом прерываний). Но для 99% прикладного кода на Go проблема "зависающих" вычислений была полностью решена.

## 3. Резюме 

* **До Go 1.14:** Планировщик был преимущественно **кооперативным**. Горутина уступала процессор на "точках безопасности" (Safe Points): при вызове функций, системных вызовах `Syscall`, работе с сетью, каналами или блокировках. Тяжелый математический `for`-цикл мог полностью парализовать ядро процессора и остановить Garbage Collector.
* **С версии Go 1.14:** Планировщик стал **асинхронно вытесняющим**. Системный монитор (`sysmon`) отслеживает горутины, работающие дольше 10 мс. Если такая найдена, ей отправляется системный сигнал `SIGURG`, который вызывает аппаратное прерывание и принудительно перехватывает управление, перенося горутину обратно в очередь выполнения.
* **Функция `runtime.Gosched()`:** По-прежнему актуальна, если программист хочет явно уступить процессорное время другой горутине, не дожидаясь срабатывания лимита в 10 мс.
