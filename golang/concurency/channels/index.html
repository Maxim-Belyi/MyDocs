channels

Знаменитый девиз языка Go гласит:

"Не общайтесь, разделяя память; вместо этого разделяйте память, общаясь." (Do not communicate by sharing memory; instead, share memory by communicating).

Вместо того чтобы две горутины пытались одновременно изменить одну переменную (что ведет к состоянию гонки и требует мьютексов), Go предлагает использовать Каналы (Channels).

Канал — это потокобезопасная труба, через которую одна горутина может отправить данные, а другая — получить.

Синтаксис и основы​

Каналы строго типизированы (в канал chan int нельзя положить строку). Создаются они через встроенную функцию make.

ch := make(chan int) // Создание канала для целых чисел

go func() {
    ch <- 42 // Отправляем данные В канал
}()

value := <-ch // Читаем данные ИЗ канала (стрелочка указывает ОТ канала)
fmt.Println(value) // 42

Буферизированные и небуферизированные каналы​

Это фундамент работы с каналами. От их типа зависит поведение горутин (блокировка).

1. Небуферизированный канал (Unbuffered)​

Создается без указания емкости: make(chan int). В таком канале нет места для хранения данных.

  • Правило отправки: Горутина, которая отправляет данные (ch <- 1), заблокируется (уснет) до тех пор, пока другая горутина не прочитает эти данные (<-ch).
  • Правило чтения: Горутина, которая читает данные, заблокируется, пока кто-нибудь не отправит данные в канал.
  • Суть: Это не просто передача данных, это строгая синхронизация. Обе горутины встречаются в одной точке времени.

2. Буферизированный канал (Buffered)​

Создается с указанием емкости: make(chan int, 3). Внутри канала появляется очередь (кольцевой буфер), в данном случае на 3 элемента.

  • Горутина-отправитель НЕ блокируется, пока в буфере есть свободное место. Она просто кладет значение и идет дальше.
  • Отправитель заблокируется только тогда, когда буфер полностью заполнится (попытка положить 4-й элемент).
  • Читатель блокируется только тогда, когда буфер пуст.

Конструкция select (Мультиплексор каналов)​

select работает как оператор switch, но только для каналов. Он позволяет одной горутине одновременно ожидать операций от нескольких каналов.

ch1 := make(chan string)
ch2 := make(chan string)

// Горутина заблокируется здесь и будет ждать,
// пока хотя бы один канал не будет готов к чтению/записи.
select {
case msg1 := <-ch1:
    fmt.Println("Получено из ch1:", msg1)
case msg2 := <-ch2:
    fmt.Println("Получено из ch2:", msg2)
}

Если одновременно готовы оба канала, select выберет один из них абсолютно случайно.

Важные паттерны с select:​

1. Неблокирующее чтение (default)​

Если добавить блок default, select не будет блокироваться. Он проверит каналы, и если ни один не готов прямо сейчас, мгновенно выполнит default.

select {
case msg := <-ch:
    fmt.Println("Получено:", msg)
default:
    fmt.Println("Канал пуст, идем дальше. Не блокируемся!")
}

2. Таймауты (Timeout)​

Как не зависнуть навсегда, если в канал никто не пишет? Использовать time.After:

select {
case res := <-resultChan:
    fmt.Println("Успех:", res)
case <-time.After(2 * time.Second): // Вернет канал, в который придет сигнал через 2 сек
    fmt.Println("Ошибка: превышено время ожидания (Таймаут)!")
}

🔥 Топ-вопросы​

Интервьюеры обожают проверять, знаешь ли ты, что вызывает panic, а что приводит к вечной блокировке (deadlock).

  1. Что будет, если прочитать из закрытого канала?
    • Паники не будет! Ты мгновенно получишь "нулевое значение" типа (zero-value). Для chan int вернется 0.
    • Как отличить реальный 0 от закрытого канала? Использовать вторую переменную: val, ok := <-ch. Если ok == false, канал закрыт и пуст.
  2. Что будет, если записать в закрытый канал?
    • ПАНИКА (panic: send on closed channel). Правило архитектуры: канал всегда закрывает та горутина, которая в него пишет!
  3. Что будет, если закрыть уже закрытый канал?
    • ПАНИКА.
  4. Что будет, если писать или читать из nil канала (неинициализированного)?
    • Горутина навсегда заблокируется (вечный сон).

Под капотом канала (Коротко для Middle+)​

Канал (структура hchan в исходниках) — это потокобезопасная структура, состоящая из:

  1. Мьютекса (Mutex): Защищает все внутренние поля канала от состояния гонки.
  2. Кольцевого буфера (Ring Buffer): Обычный массив для хранения элементов (если канал буферизированный).
  3. Двух очередей ожидания (Wait Queues): sendq и recvq (очередь ожидающих отправителей и получателей). Это связанные списки из заблокированных горутин (G).

Каналы

Для защиты общей памяти используется WaitGroup и Mutex. Но в Go есть более элегантный способ: каналы (chan). Вместо того чтобы «драться» за одну переменную, горутины передают данные друг другу - это делает код чище и избавляет от многих проблем с блокировками.

1. Что такое канал?​

Канал — это типизированная «труба», по которой горутины могут пересылать данные.

  • Типизация: В канал chan int нельзя отправить строку.
  • Синхронность: По умолчанию каналы блокирующие. Если горутина А пытается отправить данные в канал, она остановится и будет ждать, пока горутина Б их не заберет (и наоборот).
func main() {
    // Создаем канал для целых чисел
    ch := make(chan int)

    go func() {
        fmt.Println("Горутина: отправляю данные...")
        ch <- 42 // Блокировка здесь, пока кто-то не заберет
    }()

    val := <-ch // Блокировка здесь, пока кто-то не отправит
    fmt.Println("Main: получил", val)
}

2. Буферизированные каналы​

Если вы хотите, чтобы горутина не ждала получателя, можно создать буферизированный канал. Он имеет «емкость» (буфер). Отправка блокируется только тогда, когда буфер полностью заполнен.

// Канал на 2 элемента. Отправитель не будет ждать, пока получатель освободит место.
ch := make(chan int, 2) 

ch <- 1
ch <- 2
// ch <- 3 // Тут будет блокировка, так как буфер полон (size 2)

3. Паттерн «Worker Pool» (Пул рабочих)​

Это классический способ организации работы в Go. У вас есть «задача», которую должны выполнить несколько «рабочих».

func worker(id int, jobs <-chan int, results chan<- int) {
    for j := range jobs { // Читаем из канала, пока он не закроется
        fmt.Printf("Рабочий %d выполняет задачу %d\n", id, j)
        results <- j * 2
    }
}

func main() {
    jobs := make(chan int, 5)
    results := make(chan int, 5)

    // Запускаем 3 рабочих
    for w := 1; w <= 3; w++ {
        go worker(w, jobs, results)
    }

    // Раздаем 5 задач
    for j := 1; j <= 5; j++ {
        jobs <- j
    }
    close(jobs) // Важно: закрываем канал, чтобы рабочие узнали, что задач больше нет

    // Собираем результаты
    for a := 1; a <= 5; a++ {
        fmt.Println("Результат:", <-results)
    }
}

4. Паттерн «Fan-in» (Сборщик)​

Представьте, что у вас есть несколько источников данных, и вы хотите слить всё в один поток.

func producer(msg string, ch chan<- string) {
    ch <- msg
}

func main() {
    ch := make(chan string)
    
    go producer("Привет из первой горутины", ch)
    go producer("Привет из второй горутины", ch)

    // Читаем из канала дважды
    fmt.Println(<-ch)
    fmt.Println(<-ch)
}

5. Выбор с select​

Что делать, если нужно слушать сразу несколько каналов или добавить таймаут? Используйте select. Это как switch, но для операций с каналами.

select {
case msg1 := <-ch1:
    fmt.Println("Данные из 1:", msg1)
case msg2 := <-ch2:
    fmt.Println("Данные из 2:", msg2)
case <-time.After(1 * time.Second):
    fmt.Println("Таймаут: никто не прислал данные за 1 секунду")
}

Итоговая шпаргалка по каналам​

ОперацияСинтаксисЧто произойдет?
Отправкаch <- valБлокируется, пока кто-то не прочитает (или пока не освободится место в буфере).
Чтениеval := <-chБлокируется, пока кто-то не отправит данные.
Закрытиеclose(ch)Отправитель больше не может писать. Читатели получают данные, пока буфер не опустеет.
Итерацияfor v := range chЧитает данные, пока канал не будет закрыт.

Золотое правило:

  • Если данные передаются от одного потока к другому — используйте каналы.
  • Если данные — это общие настройки или счетчик, к которому часто обращаются — используйте Mutex.
  • Не закрывайте канал со стороны получателя — это всегда должен делать единственный отправитель!
