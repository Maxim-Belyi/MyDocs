# Tee-Channel (Тройник)

Иногда при построении конвейеров (Pipelines) вам нужно передать один и тот же поток данных сразу в две разные системы. Например, вы читаете события из брокера сообщений, и вам нужно:
1. Записать их в базу данных.
2. Параллельно отправить их в систему логирования/аналитики.

Вы не можете просто передать один и тот же канал `<-chan` в две разные горутины, потому что они начнут **конкурировать (Race)** за данные (Fan-Out). В результате первая горутина прочитает половину событий, а вторая — оставшуюся половину. 

Чтобы скопировать (разветвить) данные так, чтобы *обе* горутины получили *все* события, используется паттерн **Tee** (назван в честь команды `tee` в Linux, которая пишет стандартный ввод и в файл, и на экран, и похожа на Т-образный разветвитель для труб).

---

## Реализация Tee-Channel

Функция `Tee` принимает на вход один канал, а возвращает два новых независимых канала (out1, out2), в которые дублируются все сообщения.

```go
package main

import (
	"fmt"
	"time"
)

// Tee разветвляет входной канал in на два выходных: out1 и out2
func Tee(done <-chan struct{}, in <-chan interface{}) (_, _ <-chan interface{}) {
	out1 := make(chan interface{})
	out2 := make(chan interface{})

	go func() {
		defer close(out1)
		defer close(out2)

		for val := range in {
			// Локальные переменные, хранящие копии каналов.
			// Это нужно, чтобы если один канал заблокировался (никто не читает),
			// мы не заблокировали запись во второй канал. Мы будем обнулять
			// эти переменные по мере успешной записи.
			var out1Copy, out2Copy = out1, out2
			
			// Мы должны записать в ОБА канала, поэтому запускаем цикл из 2 итераций
			for i := 0; i < 2; i++ {
				select {
				case <-done:
					return
				case out1Copy <- val:
					// Запись в out1 успешна. Обнуляем копию, чтобы 
					// на следующей итерации этот case заблокировался (запись в nil-канал блокируется вечно)
					out1Copy = nil
				case out2Copy <- val:
					// Запись в out2 успешна. Обнуляем копию.
					out2Copy = nil
				}
			}
		}
	}()

	return out1, out2
}
```

### Как работает трюк с `nil`-каналами?
Ключевая магия этого паттерна кроется в поведении языка Go: **запись (и чтение) в `nil` канал навсегда блокируется**. 

Когда мы успешно отправляем значение в `out1Copy <- val`, мы присваиваем `out1Copy = nil`. На второй итерации цикла внутренний `select` уже не сможет зайти в ветку `case out1Copy <- val` (она отключена), и ему придется дождаться, пока освободится `out2Copy`. Таким образом, мы гарантируем доставку одного и того же элемента строго по одному разу в оба канала, независимо от того, кто из них читает быстрее!

## Использование

```go
func main() {
	done := make(chan struct{})
	defer close(done)

	// Входной поток данных (наш генератор из предыдущих статей)
	inStream := generator(done, 1, 2, 3)

	// Разветвляем
	out1, out2 := Tee(done, inStream)

	// Читаем из первого рукава
	go func() {
		for v1 := range out1 {
			fmt.Printf("Рукав 1: %v\n", v1)
		}
	}()

	// Читаем из второго рукава
	go func() {
		for v2 := range out2 {
			fmt.Printf("Рукав 2: %v\n", v2)
		}
	}()

	// Ждем, чтобы горутины успели отработать
	time.Sleep(1 * time.Second)
}
```

Этот паттерн критически важен, когда вам нужно реализовать **Audit Logging** (аудит лог) или скопировать трафик (Shadowing) без влияния на основной пайплайн обработки.
