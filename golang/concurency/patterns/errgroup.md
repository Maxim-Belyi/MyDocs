# ErrGroup (Группа ошибок)

Когда вы запускаете несколько независимых горутин (например, через `sync.WaitGroup`), у вас возникает огромная проблема: **как вернуть ошибки из этих горутин в основной поток?**

`WaitGroup` не умеет возвращать значения или ошибки. Обычно разработчикам приходится создавать отдельный канал `errChan <-chan error`, писать в него ошибки, а затем в основном потоке считывать их через `select`. Это делает код объемным и склонным к утечкам.

Более того, если одна из горутин упадет с ошибкой, мы, скорее всего, захотим немедленно отменить (cancel) все остальные запущенные горутины, чтобы не тратить ресурсы впустую.

Для элегантного решения этой проблемы существует официальный паттерн в расширенной библиотеке Go — пакет `golang.org/x/sync/errgroup`.

---

## Что такое ErrGroup?

**ErrGroup** — это обертка над `sync.WaitGroup` и `context.Context`, которая:
1. Запускает горутины и автоматически отслеживает их завершение.
2. Ловит **первую** вернувшуюся ошибку (от любой из горутин) и возвращает её из функции `Wait()`.
3. Автоматически **отменяет контекст** для всех остальных работающих горутин, если хотя бы одна завершилась с ошибкой.

## Использование паттерна

Для использования нужно импортировать `golang.org/x/sync/errgroup`.

### Пример: Параллельные сетевые запросы

Представьте, что нам нужно получить профиль пользователя, его баланс и историю заказов из трех разных микросервисов параллельно. Если хоть один сервис недоступен, мы отдаем ошибку 500, и нам не нужны результаты остальных сервисов.

```go
package main

import (
	"context"
	"fmt"
	"time"

	"golang.org/x/sync/errgroup"
)

// Симуляция долгого сетевого вызова
func fetchAPI(ctx context.Context, serviceName string, delay time.Duration, fail bool) error {
	select {
	case <-time.After(delay):
		if fail {
			return fmt.Errorf("ошибка в сервисе %s", serviceName)
		}
		fmt.Printf("Успешно получен ответ от %s\n", serviceName)
		return nil
	case <-ctx.Done():
		// Если контекст был отменен (из-за ошибки в другой горутине), мы прерываем работу
		fmt.Printf("Отменено выполнение %s из-за сбоя в другой горутине\n", serviceName)
		return ctx.Err()
	}
}

func main() {
	// Создаем базовый контекст (например, от входящего HTTP-запроса)
	ctx := context.Background()

	// Создаем ErrGroup. Она возвращает группу (g) и НОВЫЙ дочерний контекст (ctx).
	// Если любая горутина вернет ошибку, этот ctx будет немедленно отменен.
	g, ctx := errgroup.WithContext(ctx)

	// 1. Запускаем получение профиля (успешно, 2 сек)
	g.Go(func() error {
		return fetchAPI(ctx, "ProfileService", 2*time.Second, false)
	})

	// 2. Запускаем получение баланса (УПАДЕТ через 1 сек)
	g.Go(func() error {
		return fetchAPI(ctx, "BalanceService", 1*time.Second, true)
	})

	// 3. Запускаем получение истории (успешно, но долго - 3 сек)
	g.Go(func() error {
		return fetchAPI(ctx, "HistoryService", 3*time.Second, false)
	})

	// Ожидаем завершения всех горутин.
	// Если хоть одна вернула ошибку, Wait() вернет ПЕРВУЮ ошибку.
	if err := g.Wait(); err != nil {
		fmt.Printf("Общая ошибка: %v\n", err)
	} else {
		fmt.Println("Все сервисы ответили успешно")
	}
}
```

### Вывод этой программы:
```text
Общая ошибка: ошибка в сервисе BalanceService
Отменено выполнение ProfileService из-за сбоя в другой горутине
Отменено выполнение HistoryService из-за сбоя в другой горутине
```
Как видите, `BalanceService` упал первым (через 1 секунду). `errgroup` автоматически отменил `ctx`, и две другие горутины, которые должны были выполняться 2 и 3 секунды, немедленно прервали работу и освободили ресурсы!

## Резюме
- Если вам нужно просто дождаться горутин и не важны ошибки — используйте `sync.WaitGroup`.
- Если горутины могут вернуть ошибки, и при первой ошибке нужно прервать остальные горутины — **всегда используйте `errgroup.WithContext`**. Это "золотой стандарт" продакшен-кода на Go.
