# Каналы
Знаменитый девиз языка Go гласит: 
> *"Не общайтесь, разделяя память; вместо этого разделяйте память, общаясь."* (Do not communicate by sharing memory; instead, share memory by communicating).

Вместо того чтобы две горутины пытались одновременно изменить одну переменную (что ведет к состоянию гонки и требует мьютексов), Go предлагает использовать **Каналы (Channels)**.

**Канал** — это потокобезопасная труба, через которую одна горутина может отправить данные, а другая — получить.

### Синтаксис и основы

Каналы строго типизированы (в канал `chan int` нельзя положить строку).
Создаются они через встроенную функцию `make`.

```go
ch := make(chan int) // Создание канала для целых чисел

go func() {
    ch <- 42 // Отправляем данные В канал
}()

value := <-ch // Читаем данные ИЗ канала (стрелочка указывает ОТ канала)
fmt.Println(value) // 42
```

## Буферизированные и небуферизированные каналы

Это фундамент работы с каналами. От их типа зависит поведение горутин (блокировка).

### 1. Небуферизированный канал (Unbuffered)
Создается без указания емкости: `make(chan int)`.
В таком канале **нет места для хранения данных**.

*   **Правило отправки:** Горутина, которая отправляет данные (`ch <- 1`), **заблокируется** (уснет) до тех пор, пока другая горутина не прочитает эти данные (`<-ch`).
*   **Правило чтения:** Горутина, которая читает данные, заблокируется, пока кто-нибудь не отправит данные в канал.
*   **Суть:** Это не просто передача данных, это строгая **синхронизация**. Обе горутины встречаются в одной точке времени.

### 2. Буферизированный канал (Buffered)
Создается с указанием емкости: `make(chan int, 3)`.
Внутри канала появляется очередь (кольцевой буфер), в данном случае на 3 элемента.

*   Горутина-отправитель **НЕ блокируется**, пока в буфере есть свободное место. Она просто кладет значение и идет дальше.
*   Отправитель заблокируется только тогда, когда буфер полностью заполнится (попытка положить 4-й элемент).
*   Читатель блокируется только тогда, когда буфер пуст.

## Конструкция `select` (Мультиплексор каналов)

`select` работает как оператор `switch`, но только для каналов. Он позволяет одной горутине одновременно ожидать операций от **нескольких** каналов.

```go
ch1 := make(chan string)
ch2 := make(chan string)

// Горутина заблокируется здесь и будет ждать,
// пока хотя бы один канал не будет готов к чтению/записи.
select {
case msg1 := <-ch1:
    fmt.Println("Получено из ch1:", msg1)
case msg2 := <-ch2:
    fmt.Println("Получено из ch2:", msg2)
}
```
*Если одновременно готовы оба канала, `select` выберет один из них **абсолютно случайно**.*

### Важные паттерны с `select`:

#### 1. Неблокирующее чтение (default)
Если добавить блок `default`, `select` не будет блокироваться. Он проверит каналы, и если ни один не готов прямо сейчас, мгновенно выполнит `default`.
```go
select {
case msg := <-ch:
    fmt.Println("Получено:", msg)
default:
    fmt.Println("Канал пуст, идем дальше. Не блокируемся!")
}
```

#### 2. Таймауты (Timeout)
Как не зависнуть навсегда, если в канал никто не пишет? Использовать `time.After`:
```go
select {
case res := <-resultChan:
    fmt.Println("Успех:", res)
case <-time.After(2 * time.Second): // Вернет канал, в который придет сигнал через 2 сек
    fmt.Println("Ошибка: превышено время ожидания (Таймаут)!")
}
```

## 🔥 Топ-вопросы 

Интервьюеры обожают проверять, знаешь ли ты, что вызывает `panic`, а что приводит к вечной блокировке (`deadlock`).

1.  **Что будет, если прочитать из закрытого канала?**
    *   Паники не будет! Ты мгновенно получишь "нулевое значение" типа (zero-value). Для `chan int` вернется `0`.
    *   *Как отличить реальный 0 от закрытого канала?* Использовать вторую переменную: `val, ok := <-ch`. Если `ok == false`, канал закрыт и пуст.
2.  **Что будет, если записать в закрытый канал?**
    *   **ПАНИКА** (`panic: send on closed channel`). Правило архитектуры: канал всегда закрывает та горутина, которая в него пишет!
3.  **Что будет, если закрыть уже закрытый канал?**
    *   **ПАНИКА**.
4.  **Что будет, если писать или читать из `nil` канала (неинициализированного)?**
    *   Горутина **навсегда заблокируется** (вечный сон).

#### Под капотом канала (Коротко)
Канал (структура `hchan` в исходниках) — это потокобезопасная структура, состоящая из:
1.  **Мьютекса (Mutex):** Защищает все внутренние поля канала от состояния гонки.
2.  **Кольцевого буфера (Ring Buffer):** Обычный массив для хранения элементов (если канал буферизированный).
3.  **Двух очередей ожидания (Wait Queues):** `sendq` и `recvq` (очередь ожидающих отправителей и получателей). Это связанные списки из заблокированных горутин (G).
