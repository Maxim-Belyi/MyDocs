# 1. Горутины и Планировщик (GMP-модель)

В Go мы редко работаем с системными потоками напрямую. Вместо этого мы используем **горутины (goroutines)**. 

**Горутина** — это функция, которая выполняется конкурентно (одновременно) с другими функциями. Это не потоки операционной системы (ОС), это легковесные потоки, которыми управляет сам Go (его runtime).

Чтобы запустить горутину, достаточно написать ключевое слово `go` перед вызовом функции:

```go
func printMessage(msg string) {
    fmt.Println(msg)
}

func main() {
    // Эта функция запустится в отдельной горутине
    go printMessage("Привет из горутины!")
    
    fmt.Println("Привет из главной функции (main)!")
    
    // Плохая практика: спим, чтобы горутина успела выполниться
    time.Sleep(time.Second) 
}
```
> **Важно:** Функция `main` — это тоже горутина! Если `main` завершает свою работу, программа мгновенно закрывается, жестко убивая все остальные горутины, даже если они не закончили работу. (Для правильного ожидания используется `WaitGroup`, о котором поговорим в следующей статье).

## В чем отличие Горутины от Потока ОС? 
 *"Почему мы можем запустить 100 000 горутин, но если мы запустим 100 000 потоков ОС (в Java или C++), сервер умрет?"*

1.  **Потребление памяти (Стек):** 
    *   Потоку ОС при создании выделяется жесткий стек памяти: от **1 до 2 Мегабайт**. (100k потоков сожрут 100-200 ГБ памяти!).
    *   Горутина при создании весит всего **~2 Килобайта**. Ее стек динамический: он может увеличиваться (до 1 ГБ) и сжиматься по мере необходимости.
2.  **Стоимость переключения контекста (Context Switch):** 
    *   Когда процессор переключается между потоками ОС, он должен сохранить множество регистров, обратиться к ядру ОС (Ring 0) — это занимает микросекунды (очень долго для CPU).
    *   Переключение горутин происходит в пространстве пользователя (User Space). Go-рантайму нужно сохранить всего 3 регистра (Program Counter, Stack Pointer, DX). Это в десятки раз быстрее.
3.  **Создание и уничтожение:** 
    *   Потоки ОС просит создать операционная система — это системный вызов, медленная операция. Горутины создаются мгновенно в рантайме Go.

## Под капотом: Планировщик Go и GMP-модель (Уровень Middle/Senior)

Раз горутины — это не потоки ОС, то как они выполняются на реальном процессоре? 
Для этого в Go встроен свой собственный планировщик (Scheduler). Он работает по **модели GMP**. 

Обязательно выучите расшифровку этих трех букв:

*   **G (Goroutine):** Сама горутина. Содержит свой стек, инструкции кода и текущий статус (выполняется, спит, ждет).
*   **M (Machine):** Поток операционной системы (Thread). Именно он физически выполняет код на ядре процессора.
*   **P (Processor):** Логический процессор (Контекст). Это абстракция Go чтобы `M` (поток) мог выполнять `G` (горутину), он обязан получить доступ к `P`. 
    *   *По умолчанию количество `P` равно количеству ядер вашего физического процессора (задается переменной `GOMAXPROCS`).*

### Как они взаимодействуют?
У каждого `P` есть своя локальная очередь (Local Run Queue), в которой лежат готовые к выполнению `G` (до 256 штук). Также есть Глобальная очередь (Global Run Queue) для тех горутин, что не поместились в локальные.

1. `M` берет `P`.
2. `P` выдает ему из своей очереди `G`.
3. `M` выполняет код горутины.

### Два главных механизма Планировщика

Как планировщик решает проблемы с зависанием горутин и распределением нагрузки?

#### 1. Work Stealing (Кража работы)
Представьте, что у нас есть два логических процессора (`P1` и `P2`). `P1` быстро выполнил все горутины из своей локальной очереди, а у `P2` в очереди их еще 100. 
Вместо того чтобы простаивать, `P1` **украдет** половину горутин из очереди `P2` и начнет их выполнять! Это обеспечивает идеальный баланс баланс нагрузки на ядра процессора.

#### 2. Что происходит при блокировке (Syscall)?
Это классический вопрос: *"В горутине мы делаем долгий сетевой запрос или читаем тяжелый файл с диска (блокирующий системный вызов). Заблокирует ли это весь поток ОС и ядро?"*

**Ответ:** 
1. Если горутина (`G`) делает сетевой запрос, она не блокирует поток (`M`). Она отправляется в специальный компонент **NetPoller**, а `M` берет следующую горутину.
2. Если горутина делает тяжелый дисковый вызов, то поток ОС (`M`) действительно блокируется. 
3. *Но планировщик Go не спит!* Заметив, что `M1` заблокировался, он **отцепляет** процессор `P` от зависшего `M1` и передает его другому, свободному потоку `M2` (или создает новый поток). В итоге `P` продолжает выполнять другие горутины, пока `M1` ждет жесткий диск. Когда диск ответит, горутина вернется в очередь, а поток `M1` отправится "в запас" или умрет.

**Пример:**

Представь кухню ресторана. Есть два повара (P1 и P2).

* У P1 в списке 2 заказа (две горутины). Он их быстро приготовил.
* У P2 в списке 50 заказов (он попал на банкет).
* Проблема: P1 стоит без дела, а P2 зашивается.
* Решение (Work Stealing): P1 идет к P2, смотрит в его список и забирает (крадет) ровно половину (255/2 или в нашем случае 25 заказов).

Почему это важно?
Это минимизирует простои ядер процессора. Перераспределение происходит молниеносно внутри программы в Ring 3 (User Mode), без обращений к ядру ОС.

### Шпаргалка для собеседования
1.  **Горутины** легкие (~2KB), их можно запускать сотнями тысяч.
2.  Они не привязаны жестко к потокам ОС (много горутин мультиплексируются на меньшее количество потоков).
3.  **G** = Горутина, **M** = Поток ОС, **P** = Контекст (Очередь выполнения).
4.  Функция `main` не ждет завершения остальных горутин по умолчанию.
