# Внутреннее устройство каналов (hchan)

Каналы в Go — это инструмент для синхронизации и обмена данными между горутинами. На уровне исходного кода рантайма Go канал представляет собой сложную структуру `hchan` (сокращение от header of channel), написанную на языке C (в ранних версиях) и Go. Разбор этой структуры часто встречается на собеседованиях уровня Middle+/Senior.

---

## 1. Структура `hchan`

Физически канал — это указатель на структуру `hchan`, расположенную в куче (heap). Именно поэтому мы создаем каналы через `make(chan T)` и можем безопасно передавать их в функции по значению (копируется только указатель).

Основные поля структуры `hchan` (`src/runtime/chan.go`):

```go
type hchan struct {
	qcount   uint           // Текущее количество элементов в буфере
	dataqsiz uint           // Размер кругового буфера (задается при make)
	buf      unsafe.Pointer // Указатель на массив памяти кольцевого буфера
	elemsize uint16         // Размер одного элемента в байтах (зависит от типа T)
	closed   uint32         // Флаг состояния: 0 (открыт), 1 (закрыт)
	elemtype *_type         // Указатель на информацию о типе передаваемых элементов
	sendx    uint           // Индекс (позиция) для следующей отправки (записи)
	recvx    uint           // Индекс (позиция) для следующего получения (чтения)
	recvq    waitq          // Двусвязный список горутин, ожидающих чтения (receive queue)
	sendq    waitq          // Двусвязный список горутин, ожидающих записи (send queue)

	lock mutex              // Мьютекс для защиты всех полей hchan от состояния гонки
}
```

### Ключевые компоненты:
1. **Кольцевой буфер (`buf`, `sendx`, `recvx`):** Обычный массив фиксированного размера. Если буфер заполняется до конца, указатель записи (`sendx`) перескакивает в начало (индекс 0), перезаписывая прочитанные ячейки.
2. **Очереди ожидания (`recvq`, `sendq`):** Очереди типа `waitq`. Это связанные списки структур `sudog`. Структура `sudog` (pseudo-G) оборачивает заблокированную горутину (`G`) и содержит указатель на элемент памяти (переменную в стеке горутины), который эта горутина хочет отправить или получить.
3. **Глобальная блокировка (`lock`):** Мьютекс на уровне рантайма. Любая операция с каналом (отправка, чтение, закрытие) начинается с захвата этого мьютекса `lock.Lock()` и заканчивается его освобождением. Именно поэтому каналы потокобезопасны.

## 2. Механика работы: Сценарии под капотом

Рассмотрим, как физически передаются данные при вызове `ch <- data` (метод `chansend`) и `<-ch` (метод `chanrecv`).

### Сценарий А: Буферизированный канал (есть свободное место)
Горутина-отправитель (G1) выполняет `ch <- data`. В буфере есть свободная ячейка.

1. G1 захватывает `lock`.
2. Копирует `data` из своего стека прямо в ячейку массива `buf` по индексу `sendx` (используется функция `typedmemmove`).
3. Увеличивает `sendx` на 1.
4. Увеличивает `qcount` на 1.
5. Отпускает `lock`.
*Переключение контекста не происходит, G1 продолжает работу.*

### Сценарий Б: Канал полон (или небуферизированный) и нет получателя
G1 пытается отправить `data`, но места нет.

1. G1 захватывает `lock`.
2. Рантайм создает структуру `sudog`, кладет в нее ссылку на саму G1 и ссылку на передаваемый элемент `data`.
3. Этот `sudog` добавляется в хвост очереди ожидающих отправителей `sendq`.
4. G1 отпускает `lock`.
5. Вызывается системная функция `gopark()`. Горутина G1 переводится в статус `Waiting` (спит). Логический процессор `P` берет другую горутину.

### Сценарий В: Передача напрямую (Direct Send)
Это оптимизация производительности рантайма Go, позволяющая избежать лишнего копирования и захвата мьютексов.
G2 вызывает `<-ch` на пустом канале. Она засыпает и попадает в очередь `recvq`. Затем G1 вызывает `ch <- data`.

1. G1 захватывает `lock`.
2. Замечает, что в `recvq` уже спит горутина G2.
3. Рантайм **НЕ кладет данные в `buf`**. Вместо этого G1 копирует данные `data` **напрямую из своего стека в стек спящей горутины G2**.
4. G1 отпускает `lock`.
5. G1 вызывает функцию `goready(G2)`. Спящая горутина G2 переводится в статус `Runnable` и помещается в локальную очередь процессора `P`.

### Сценарий Г: Закрытие канала (`close`)
1. Захватывается `lock`.
2. Флаг `closed` устанавливается в `1`.
3. Все горутины, спящие в очереди `recvq` (ждущие данных), получают `zero value` (например, `0` для int, `""` для string) и флаг `ok = false`. Рантайм вызывает для них `goready`.
4. Все горутины, спящие в очереди `sendq` (ждущие места), пробуждаются с паникой (`panic: send on closed channel`).
5. `lock` отпускается.

---

## 3. Резюме 

* **Канал — это очередь с блокировкой.** Он содержит мьютекс (`lock`), кольцевой буфер данных (`buf`) и две очереди заблокированных горутин (`sendq`, `recvq`).
* Данные в канал всегда **передаются по значению** (физически копируются в кучу канала, а затем в стек получателя).
* Оптимизация **Direct Send** позволяет горутине-отправителю копировать данные напрямую в стек горутине-получателю в обход буфера, если получатель уже спит в `recvq`.
* **Горутины паркуются (засыпают)** через функцию `gopark` и пробуждаются через `goready`. Пока горутина спит в `waitq` канала, она не потребляет процессорное время.
