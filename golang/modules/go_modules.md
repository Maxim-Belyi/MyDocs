# Управление зависимостями: Go Modules

В Go управление зависимостями встроено в сам инструмент командной строки. Система **Go Modules** — это стандарт де-факто, который позволяет фиксировать версии библиотек и гарантировать воспроизводимость сборки проекта на любой машине.

---

## 1. Основные файлы: go.mod и go.sum

Когда вы инициализируете проект командой `go mod init <module_name>`, создаются два ключевых файла.

### go.mod
Это "паспорт" вашего проекта. В нем указаны:
*   Имя модуля (путь, по которому его будут импортировать другие).
*   Версия Go, для которой написан код.
*   Список зависимостей с их версиями.

```go
module github.com/user/myproject

go 1.22

require (
    github.com/google/uuid v1.6.0
    github.com/lib/pq v1.10.9 // indirect
)
```
> **// indirect** означает, что ваша программа не использует эту библиотеку напрямую, но она нужна одной из ваших зависимостей.

### go.sum
Это файл с контрольными суммами (хешами) всех зависимостей.
*   **Зачем нужен?** Чтобы гарантировать, что скачанная библиотека не была подменена или изменена злоумышленниками.
*   **Правило:** Никогда не редактируйте этот файл вручную, но обязательно фиксируйте его в Git.

---

## 2. Команда go mod tidy

Это самая часто используемая команда. Она делает "уборку" в вашем проекте:
1.  Добавляет в `go.mod` модули, которые вы импортировали в коде, но еще не скачали.
2.  Удаляет из `go.mod` лишние зависимости, которые больше не используются в коде.
3.  Обновляет `go.sum`.

**Совет:** Запускайте `go mod tidy` перед каждым коммитом.

---

## 3. Управление мажорными версиями (v2+)

В Go соблюдается принцип семантического версионирования (SemVer). Если вы выпускаете ломающие изменения, вы обязаны сменить мажорную версию.

В Go v2 и выше — это **другой модуль**. Путь импорта должен содержать номер версии в конце:

```go
// go.mod для второй версии
module github.com/user/myproject/v2

go 1.22
```

**Зачем это нужно?** Это позволяет одному проекту использовать одновременно две разные версии одной и той же библиотеки (например, v1 и v2) без конфликтов имен.

---

## 4. Приватные репозитории: GOPRIVATE

По умолчанию Go скачивает модули через публичный прокси-сервер Google (`proxy.golang.org`). Но он не сможет скачать ваш приватный код из GitLab или корпоративного GitHub.

Чтобы это исправить, используйте переменную окружения `GOPRIVATE`:

```bash
go env -w GOPRIVATE=github.com/my-company/*
```

Это скажет Go: "Для всех репозиториев компании не используй публичный прокси и не проверяй контрольные суммы в публичной базе, качай напрямую через Git".

## 5. Go Workspaces (go.work)

Введено в Go 1.18. Позволяет работать над несколькими модулями одновременно, когда один модуль зависит от локальных изменений в другом, которые еще не залиты в репозиторий.

Создайте файл `go.work` в корневой папке:

```go
go 1.22

use (
    ./my-app
    ./my-library
)
```

Теперь, когда вы запускаете `my-app`, Go увидит, что `my-library` лежит рядом на диске, и будет использовать локальный код вместо того, чтобы скачивать его из интернета. Это гораздо удобнее, чем старый способ с директивой `replace`.

---

## Итог: Чек-лист по работе с модулями

1.  **Инициализация:** `go mod init <name>`.
2.  **Добавление новой библиотеки:** `go get <package_path>@latest`.
3.  **Очистка проекта:** `go mod tidy`.
4.  **Проверка обновлений:** `go list -u -m all`.
5.  **Локальная разработка нескольких модулей:** используйте `go.work`.
