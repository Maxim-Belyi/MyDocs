# Стандартная структура проекта в Go (Standard Project Layout)

В экосистеме языка Go нет жестко закрепленной на уровне компилятора структуры каталогов, однако сообществом сформирован стандарт `golang-standards/project-layout`. Он применяется для построения масштабируемых микросервисов и монолитных приложений, четко разграничивая публичные интерфейсы, приватную бизнес-логику и точки входа.

---

## 1. Общий обзор типового дерева директорий

```text
my-project/
├── cmd/
│   └── app/
│       └── main.go
├── internal/
│   ├── domain/
│   ├── service/
│   ├── repository/
│   └── transport/
│       └── http/
├── pkg/
│   └── logger/
├── api/
│   └── openapi/
│       └── swagger.yaml
├── configs/
│   └── config.yaml
├── migrations/
│   └── 000001_init.up.sql
├── scripts/
├── build/
│   └── Dockerfile
├── go.mod
├── go.sum
└── Makefile
```

---

## 2. Назначение ключевых директорий

### `cmd/`
Точка входа в приложение (Main Applications).
* **Что хранить:** Каталог для каждого исполняемого файла проекта (например, `cmd/api-server/main.go`, `cmd/worker/main.go`, `cmd/cli/main.go`).
* **Правило:** Файлы `main.go` внутри этой директории должны оставаться максимально тонкими. Их единственная задача — считать конфигурацию, инициализировав логгер и основные сервисы, запустить приложение (связать зависимости через Dependency Injection) и корректно обработать сигнал завершения (Graceful Shutdown). Никакой бизнес-логики здесь быть не должно.

### `internal/`
Приватный код приложения и бизнес-логика.
* **Что хранить:** Все пакеты, которые специфичны исключительно для данного конкретного сервиса и не предназначены для повторного использования другими внешними проектами.
* **Системное ограничение Go:** Компилятор Go на уровне самого языка запрещает импортировать пакеты из директории `internal/` внешним модулям (другим проектам или сервисам). Попытка выполнить импорт из чужого `internal/` вызовет ошибку компиляции.
* **Типовая структура внутри `internal/` (Чистая архитектура):**
  * `domain/` (или `model/`): Базовые структуры данных и интерфейсы бизнес-сущностей без внешних зависимостей.
  * `service/` (или `usecase/`): Прикладная бизнес-логика, оркестрирующая работу репозиториев и внешних клиентов.
  * `repository/` (или `storage/`): Слой взаимодействия с базами данных (SQL, NoSQL, кэш).
  * `transport/` (или `handler/`, `controller/`): Обработчики входящих запросов (HTTP REST, gRPC, консьюмеры RabbitMQ/Kafka).

### `pkg/`
Публичный библиотечный код.
* **Что хранить:** Пакеты, которые не содержат специфичной бизнес-логики конкретного сервиса и полностью безопасны для переиспользования другими проектами команды или внешними разработчиками.
* **Примеры:** Кастомные обертки над логгерами (`pkg/logger`), утилиты для шифрования (`pkg/crypto`), клиенты для фирменных API (`pkg/apiclient`).
* **Важное правило:** Если вы сомневаетесь, куда поместить код — в `internal/` или `pkg/`, всегда помещайте его в `internal/`. Перенести код из `internal/` в `pkg/` в будущем безопасно, тогда как обратное действие может сломать внешние зависимости.

## 3. Вспомогательные директории

* `api/`: Протоколы описания API: спецификации OpenAPI/Swagger (`.json`/`.yaml`), файлы Protocol Buffers (`.proto`) и сгенерированные по ним gRPC-клиенты/серверы.
* `configs/`: Файлы конфигураций (YAML, TOML, JSON) или шаблоны `.env.example` для локального развертывания.
* `migrations/`: SQL-скрипты миграций базы данных (`up` и `down` файлы) для инструментов `golang-migrate` или `goose`.
* `scripts/`: Скрипты автоматизации сборки, генерации моков, линтинга или CI/CD пайплайнов (`.sh`, `.py`).
* `build/`: Конфигурации для сборки и развертывания (Dockerfiles, манифесты Kubernetes `k8s/`, Helm-чарты).
