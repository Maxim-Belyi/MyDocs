Рефлексия (Reflection) и кодогенерация

Go — статически типизированный язык, но иногда нам нужно работать с типами, которые неизвестны на этапе написания кода (например, при написании JSON-парсера или ORM). Для этого в Go есть пакет reflect.

1. Основы рефлексии: Type и Value​

Рефлексия в Go строится вокруг двух главных понятий:

  1. reflect.Type: информация о типе (название, методы, поля структуры).
  2. reflect.Value: информация о конкретном значении (само число, строка или экземпляр структуры).
import "reflect"

func main() {
    x := 42
    t := reflect.TypeOf(x)  // int
    v := reflect.ValueOf(x) // 42
    
    fmt.Printf("Тип: %v, Значение: %v\n", t, v)
}

2. Работа со структурами и тегами​

Самый частый сценарий использования рефлексии — чтение тегов структур (struct tags).

type User struct {
    Name  string `json:"user_name" db:"name"`
    Email string `json:"email"`
}

func main() {
    u := User{Name: "Ivan"}
    t := reflect.TypeOf(u)
    
    field, _ := t.FieldByName("Name")
    fmt.Println("JSON тег поля Name:", field.Tag.Get("json")) // user_name
}

Именно так работают библиотеки encoding/json и sqlx — они смотрят на теги и решают, в какую колонку БД или поле JSON положить данные.

3. Изменение значений через рефлексию​

Чтобы изменить значение через reflect, вы должны передать указатель на переменную, иначе рефлексия получит лишь копию и не сможет её изменить.

func main() {
    x := 10
    v := reflect.ValueOf(&x).Elem() // Elem() переходит от указателя к самому значению
    
    if v.CanSet() {
        v.SetInt(20)
    }
    fmt.Println(x) // 20
}

4. Почему рефлексия — это больно?​

  1. Производительность: Рефлексия в 10-100 раз медленнее обычного кода. Каждое обращение требует проверки типов в рантайме.
  2. Отсутствие проверок компилятора: Если вы ошибетесь в имени поля или типе, программа упадет с panic во время работы, а не во время сборки.

Совет: Избегайте рефлексии везде, где можно обойтись интерфейсами или дженериками.

5. Кодогенерация (go generate)​

Поскольку рефлексия медленная, в мире Go популярен подход кодогенерации. Вместо того чтобы изучать типы в рантайме, мы запускаем скрипт, который генерирует Go-код специально под наши типы ещё до компиляции.

Пример: инструмент stringer​

Если у вас есть перечисление (enum), вы хотите, чтобы оно красиво выводилось в логах. Вместо рефлексии используется генератор.

// Добавляем магический комментарий:
//go:generate stringer -type=Pill

type Pill int

const (
    Placebo Pill = iota
    Aspirin
    Ibuprofen
)

После запуска команды go generate ./... в папке появится файл pill_string.go с методом String(). Это работает максимально быстро, так как это обычный статически сгенерированный код.

Итог: Что выбрать?​

ИнструментПлюсыМинусы
ИнтерфейсыСамый идиоматичный и безопасный путь.Нужно заранее знать поведение.
ДженерикиБыстро (нет оверхеда в рантайме), типобезопасно.Иногда не хватает гибкости для метапрограммирования.
РефлексияМаксимальная гибкость, работает с любыми данными.Медленно, опасно (паники).
КодогенерацияБыстро как обычный код, ошибки видны при компиляции.Усложняет процесс сборки (нужно не забывать генерировать).

Главный вывод: Используйте рефлексию только для написания универсальных библиотек (сериализаторы, драйверы БД). В бизнес-логике приложения ей не место.
