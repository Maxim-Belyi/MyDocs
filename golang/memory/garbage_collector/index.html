Управление памятью и Garbage Collector (GC)

Go — язык с автоматическим управлением памятью. Разработчику не нужно вручную вызывать malloc и free, как в C. Вместо этого работает Garbage Collector (GC) — фоновый процесс, который автоматически находит и освобождает неиспользуемую память. Но чтобы писать быстрый код, нужно понимать, как это работает изнутри.

1. Стек (Stack) vs Куча (Heap)​

Когда вы создаете переменную в Go, она может оказаться в одном из двух мест в памяти.

Стек (Stack)​

  • Что это: Область памяти, выделяемая каждой горутине. Имеет небольшой начальный размер (обычно 8 КБ) и растет динамически.
  • Скорость: Операции очень быстрые. Это просто сдвиг одного указателя (stack pointer).
  • Жизнь переменной: Переменная автоматически уничтожается, когда функция возвращает управление. Никакого GC не нужно.

Куча (Heap)​

  • Что это: Общая область памяти для всей программы. Управляется Garbage Collector'ом.
  • Скорость: Аллокация медленнее, требует поиска свободного блока. Нагружает GC.
  • Жизнь переменной: Живет до тех пор, пока есть хотя бы одна ссылка на неё из "живого" кода.

2. Escape Analysis (Анализ Побегов)​

Именно компилятор решает, куда положить каждую переменную — в стек или в кучу. Этот анализ называется Escape Analysis.

Переменная "убегает" (escapes) в кучу, если:

  1. Её адрес (&) передается вовне функции.
  2. Она кладется в интерфейс (any).
  3. Её размер неизвестен на этапе компиляции.
  4. Она слишком большая, чтобы уместиться на стеке.
func createOnStack() int {
    x := 42    // x будет на стеке — никто не берет её адрес
    return x   // Возвращаем значение (копию)
}

func createOnHeap() *int {
    x := 42    // x "убегает" в кучу...
    return &x  // ...потому что мы возвращаем АДРЕС локальной переменной
}

Как проверить, куда ушла переменная?​

go build -gcflags="-m" ./...

Компилятор выведет все решения Escape Analysis, например: x escapes to heap.

3. Как работает Garbage Collector в Go​

GC в Go основан на алгоритме Tri-color Mark and Sweep (Трехцветная маркировка и очистка). Он работает параллельно с вашей программой (concurrent), стараясь минимизировать паузы.

Три цвета:​

  • ⚪ Белый: Объект еще не проверен GC. В конце цикла все белые объекты — мусор и будут удалены.
  • 🩶 Серый: Объект достижим, но его "дочерние" объекты ещё не проверены. Рабочая очередь GC.
  • ⚫ Черный: Объект и все его дочерние объекты проверены. Этот объект точно нужен — не удалять.

Фазы работы GC:​

1. Mark Start (STW): Самая первая пауза (Stop The World — программа останавливается). GC сканирует корневые объекты (глобальные переменные, стеки горутин) и красит их в серый. Пауза очень короткая (< 1 мс).

2. Concurrent Mark (параллельно): GC запускает горутины-маркировщики, которые обходят все серые объекты, красят их в черный и добавляют их дочерние объекты в серую очередь. Ваша программа в это время продолжает работать параллельно!

3. Mark Termination (STW): Вторая, финальная пауза. GC проверяет, что маркировка завершена корректно (используется барьер записи — Write Barrier).

4. Concurrent Sweep: Параллельно с работой программы, GC проходит по памяти и освобождает все белые (недостижимые) объекты.

4. sync.Pool: Как снизить нагрузку на GC​

sync.Pool — это пул переиспользуемых объектов. Вместо того чтобы создавать новый объект и выделять под него память (аллокация → нагрузка на GC), вы берете готовый из пула, используете и возвращаете назад.

Идеально подходит для объектов с коротким временем жизни, которые создаются очень часто (например, буферы в HTTP-обработчиках).

var bufPool = sync.Pool{
    // New вызывается, когда пул пуст
    New: func() any {
        return new(bytes.Buffer)
    },
}

func processRequest(data string) string {
    // Берем буфер из пула вместо создания нового
    buf := bufPool.Get().(*bytes.Buffer)
    buf.Reset() // Важно: всегда очищайте объект перед использованием!
    
    defer bufPool.Put(buf) // Возвращаем объект в пул после завершения
    
    buf.WriteString(data)
    return buf.String()
}

Важно: Не сохраняйте ссылки на объекты из пула дольше, чем нужно. GC может очистить пул в любой момент между двумя циклами сборки мусора — это допустимое поведение.

5. Настройка GC через GOGC​

Переменная окружения GOGC контролирует агрессивность Garbage Collector'а.

  • По умолчанию: GOGC=100 — это означает, что GC запустится, когда объем живой кучи вырастет на 100% (в 2 раза) относительно размера после прошлой сборки.
  • GOGC=200 — GC будет запускаться реже (когда куча вырастет в 3 раза). Меньше пауз, но больше потребление памяти.
  • GOGC=off — отключить GC полностью (используется только в очень специфичных сценариях).

Начиная с Go 1.19 можно также использовать runtime/debug.SetMemoryLimit() для жесткого ограничения потребления памяти.

Итог: Практические советы​

  1. Не злоупотребляйте any/interface{} в горячих путях (hot paths) — это приводит к лишним аллокациям в куче.
  2. Используйте sync.Pool для часто создаваемых краткоживущих объектов (буферы, временные структуры).
  3. Запускайте go build -gcflags="-m", чтобы видеть, куда компилятор отправляет ваши переменные.
  4. Предаллоцируйте слайсы через make([]T, 0, n), если знаете примерный размер — это избегает многократных переаллокаций.
  5. Избегайте утечек горутин — горутина, которая никогда не завершится, держит весь свой стек в памяти.
