# NoSQL: Key-Value хранилища (Redis, Memcached, etcd)

**Key-Value (Ключ-Значение)** — это самый простой тип баз данных. Если реляционные БД (SQL) можно сравнить с таблицами Excel, то Key-Value база — это **огромная `map[string]any`**, которая работает как отдельный сервер.

В ней нет таблиц, колонок или связей (Foreign Keys). Каждая запись состоит ровно из двух частей:
1.  **Ключ:** Уникальный идентификатор (обычно строка, например, `user:1024:session`).
2.  **Значение:** Данные, привязанные к ключу. Это может быть строка, число, JSON-документ, бинарные данные (картинка) или специализированные структуры (списки, множества).

### Главное преимущество
Поскольку под капотом используется хеш-таблица, чтение и запись по ключу происходят со сложностью **$O(1)$** (мгновенно), вне зависимости от того, лежит в базе 100 записей или 1 миллиард.

## 1. Redis 

**Redis** (Remote Dictionary Server) — это In-Memory база данных. Это значит, что все данные она хранит в **оперативной памяти (RAM)**, а не на жестком диске. Благодаря этому Redis работает невероятно быстро (миллионы операций в секунду).

### Главные особенности Redis:
1.  **Однопоточность (Single-threaded):** 
    Само выполнение команд в Redis происходит в **один поток**.
    *Почему это гениально?* Вам не нужны блокировки (мьютексы) при изменении данных. Если два клиента одновременно отправят команду `INCR likes_count` (увеличить счетчик), Redis выполнит их строго по очереди. Состояние гонки (Data Race) на уровне команд исключено!
2.  **Не только строки:** 
    В качестве значений Redis поддерживает списки (Lists), множества (Sets), хеши (Hashes) и даже гео-данные.

### Как Redis не теряет данные при перезагрузке? (Persistence)
Это частый вопрос на собеседовании: *"Если Redis хранит всё в RAM, что будет, если выдернуть сервер из розетки?"*
У Redis есть два механизма сохранения данных на диск:
*   **RDB (Redis Database Backup):** Делает полные "снимки" (снапшоты) всей памяти на диск с определенной периодичностью (например, раз в 5 минут). *Плюс:* быстрая загрузка. *Минус:* можно потерять данные за последние 5 минут.
*   **AOF (Append Only File):** Каждая команда на изменение данных (SET, INCR) сразу записывается (append) в конец специального файла на диске. *Плюс:* максимальная надежность. *Минус:* файл быстро растет, восстановление занимает больше времени.
*(Часто включают оба механизма одновременно).*

## 2. Что происходит, когда заканчивается память? (Eviction Policies)

Оперативная память дорогая и ограничена. Что сделает Redis, если вы выделили ему 2 ГБ памяти, и они заполнились, а вы пытаетесь записать новый ключ?

Настройка поведения при нехватке памяти называется **Eviction Policy (Политика вытеснения)**. Основные варианты:
*   **Noeviction:** Вернуть ошибку. Ничего не удалять. (По умолчанию).
*   **Allkeys-LRU (Least Recently Used):** Удалить те ключи, к которым **дольше всего не обращались** (самые старые неиспользуемые). Это самая популярная политика для кэширования!
*   **Allkeys-LFU (Least Frequently Used):** Удалить ключи, которые запрашиваются **реже всего**.
*   **Volatile-TTL:** Удалить только те ключи, у которых истекает время жизни (TTL - Time To Live).

> **Интервью-совет:** Если вас просят спроектировать систему кэширования (System Design), всегда упоминайте использование **Redis с политикой LRU и установкой TTL (времени жизни)** для ключей.

## 3. Основные паттерны использования Key-Value

Где именно реляционная БД не справляется, и мы берем Key-Value:

1.  **Кэширование (Caching):** Самый частый кейс. Если запрос к SQL-базе занимает 500мс, мы сохраняем результат в Redis (ключ: `sql_query_hash`, значение: `JSON`). Следующий запрос вернет данные из Redis за 1мс.
2.  **Хранение сессий (Session Store):** Сохранение токенов авторизации пользователей. Ключ: `session_token`, Значение: `user_id`. (Удобно использовать TTL, чтобы сессия сама "сгорела" через 24 часа).
3.  **Счетчики в реальном времени (Rate Limiting / Views):** Подсчет просмотров видео или ограничение "не более 10 запросов в секунду от одного IP". SQL-база от таких частых апдейтов просто "ляжет", а атомарная операция `INCR` в Redis справится играючи.
4.  **Распределенные блокировки (Distributed Locks):** Когда у вас 10 серверов с Go-приложением, и им нужно договориться, кто будет выполнять задачу, они могут использовать механизм `SETNX` (Set if Not eXists) в Redis.

## 4. Бонус для Go-разработчика: `etcd`

В мире микросервисов (и особенно в Kubernetes) правит бал другая Key-Value база — **etcd**. Она написана на Go!

*   **Чем отличается от Redis?** Redis создан для огромной скорости и теряет часть строгой согласованности при кластеризации. **etcd** создан для **абсолютной надежности и строгой консистентности** в распределенных системах.
*   **Как работает?** Использует алгоритм консенсуса **Raft**. Даже если половина серверов кластера сгорит, оставшиеся узлы точно будут знать самое актуальное состояние данных.
*   **Где применяется?** Для хранения конфигураций микросервисов, Service Discovery (обнаружения сервисов) и координации работы кластера. Kubernetes хранит всё свое состояние именно в `etcd`.

### Шпаргалка: SQL vs Key-Value

| Характеристика | Реляционные БД (PostgreSQL) | Key-Value (Redis) |
| :--- | :--- | :--- |
| **Структура** | Таблицы, колонки, строгая схема | Ключ -> Значение, схемы нет |
| **Сложность поиска** | $O(\log n)$ (По индексу B-Tree) | $O(1)$ (По хеш-таблице) |
| **Где хранит** | На жестком диске (SSD/HDD) | В оперативной памяти (RAM) |
| **Связи (JOIN)** | Поддерживает | Невозможно (нужно делать ручками в коде) |
| **Применение** | Хранение основного стейта приложения, финансы, юзеры | Кэш, сессии, счетчики, быстрые временные данные |
