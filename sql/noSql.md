# NoSQL: Документоориентированные БД (MongoDB)

Документоориентированная база данных хранит информацию не в плоских таблицах со строками и столбцами, а в виде **иерархических документов**, чаще всего в формате, похожем на JSON. 

### Главные концепции (Словарь):
*   **База данных (Database):** То же самое, что и в SQL.
*   **Коллекция (Collection):** Аналог *Таблицы* в SQL. Группирует документы.
*   **Документ (Document):** Аналог *Строки* в SQL. Сам объект с данными.
*   **Поле (Field):** Аналог *Столбца* в SQL. Ключ внутри документа.

Главное отличие: документы внутри одной коллекции **не обязаны иметь одинаковую структуру**. У одного пользователя может быть поле `age`, у другого — нет, а у третьего вместо возраста будет массив `hobbies`. Это называется **динамической схемой (Schemaless)**.

```json
{
  "_id": "507f1f77bcf86cd799439011",
  "name": "Иван Иванов",
  "email": "ivan@example.com",
  "address": {
    "city": "Москва",
    "street": "Ленина, 10"
  },
  "orders": [
    {
      "order_id": "ORD-101",
      "date": "2023-10-27T10:00:00Z",
      "total": 5500,
      "items": ["Наушники", "Чехол"]
    },
    {
      "order_id": "ORD-202",
      "total": 1200,
      "items": ["Зарядка"]
    }
  ],
  "is_premium": true
}
```

## 1. Под капотом: BSON (Binary JSON)

*"Монго хранит данные прямо в виде текста JSON?"*
**Ответ: Нет.** MongoDB использует формат **BSON** (Binary JSON).

Почему не обычный JSON?
1.  **Типы данных:** В текстовом JSON нет разницы между целым числом (`int`) и числом с плавающей точкой (`float`), нет типа "Дата" или "Бинарные данные" (например, для хранения картинки). BSON поддерживает строгую типизацию.
2.  **Скорость и Индексы:** BSON хранит длину документов и полей. Чтобы найти нужное поле, базе не нужно парсить весь текстовый документ от начала до конца (как это было бы с JSON), она может быстро "перепрыгнуть" к нужному байту.
3.  **_id (ObjectID):** Каждому документу автоматически присваивается уникальный первичный ключ `_id` типа `ObjectID`. Это 12-байтное число, которое генерируется драйвером на стороне приложения и включает в себя timestamp (время создания), поэтому документы в Монго по умолчанию отсортированы по времени добавления!

## 2. Моделирование данных: Встраивание vs Ссылки

Это **самый важный вопрос** по MongoDB на уровне проектирования (System Design). В SQL мы всегда нормализуем данные (разносим по разным таблицам и делаем `JOIN`). В MongoDB подход другой.

### Подход 1: Встраивание (Embedding / Денормализация)
Мы храним связанные данные прямо внутри одного документа в виде вложенных объектов или массивов.
```json
{
  "_id": "user_1",
  "name": "Alex",
  "addresses":[
    {"city": "Moscow", "street": "Lenina"},
    {"city": "Kazan", "street": "Baumana"}
  ]
}
```
*   **Плюс:** Невероятная скорость. Мы достаем пользователя и все его адреса за один запрос к диску ($O(1)$).
*   **Минус:** Ограничение на размер документа в MongoDB — **16 Мегабайт**. Если массив адресов будет расти бесконечно, документ превысит лимит.

### Подход 2: Ссылки (Referencing / Нормализация)
Аналог связей в SQL. Мы храним только `ID` связанных документов.
```json
// Коллекция users
{
  "_id": "user_1",
  "name": "Alex",
  "address_ids": ["addr_1", "addr_2"]
}
```
*   **Плюс:** Подходит для связи "Один-ко-Многим", если дочерних элементов тысячи или миллионы (например, комментарии к популярному посту).
*   **Минус:** Чтобы собрать полные данные, приложению придется сделать несколько запросов к БД или использовать медленный оператор `$lookup` (аналог JOIN в MongoDB).

> **Интервью-совет:** Правило большого пальца для MongoDB: «Данные, которые читаются вместе, должны храниться вместе». Если вы при каждом запросе пользователя показываете его адреса — используйте *Встраивание*. Если адреса нужны редко — *Ссылки*.

## 3. Миф об отсутствии транзакций (ACID)

*"Поддерживает ли MongoDB транзакции?"*

Многие разработчики, читавшие старые статьи, ответят "Нет, только на уровне одного документа". **Это ошибка!**
*   Действительно, исторически MongoDB гарантировала ACID только при изменении **одного документа**.
*   **Начиная с версии 4.0 (2018 год)** MongoDB полноценно поддерживает распределенные **многодокументные транзакции** (Multi-document ACID transactions). Вы можете безопасно обновлять данные в разных коллекциях с возможностью отката (`Rollback`).

Но злоупотреблять ими не стоит: транзакции в MongoDB работают медленнее, чем в PostgreSQL, так как архитектура изначально строилась под денормализованные (встроенные) данные.

## 4. Масштабирование: Replica Set и Sharding (Уровень Middle)

MongoDB изначально создавалась для огромных объемов данных (Big Data), поэтому в нее встроены мощные механизмы масштабирования:

1.  **Replica Set (Отказоустойчивость):** Данные синхронно или асинхронно копируются с основного сервера (Primary) на запасные (Secondary). Если Primary падает, один из Secondary автоматически становится новым Primary за пару секунд. Это защищает от потери данных.
2.  **Sharding (Горизонтальное масштабирование):** Если данных становится так много, что они не помещаются на один сервер (например, 10 Терабайт), MongoDB может "разрезать" коллекцию (по определенному ключу) и разложить куски на несколько разных серверов. Для приложения это выглядит как одна гигантская база. Реляционные БД (Postgres) делают это «из коробки» с огромным трудом, а для MongoDB — это родная фича.

### Шпаргалка: PostgreSQL vs MongoDB

| Характеристика | PostgreSQL (SQL) | MongoDB (NoSQL) |
| :--- | :--- | :--- |
| **Схема (Schema)** | Жесткая (надо создавать колонки заранее) | Динамическая (можно писать что угодно) |
| **Связи** | JOIN-ы, Foreign Keys (отличная поддержка) | Встраивание, `$lookup` (медленные связи) |
| **Формат данных** | Строки и Столбцы | Иерархический BSON (JSON) |
| **Масштабирование** | Вертикальное (покупаем более мощный сервер) | Горизонтальное (добавляем больше дешевых серверов — Sharding) |
| **Идеально для...** | Финансы, биллинг, четко структурированные данные | Каталоги товаров, логи, IoT, профили с гибкими настройками |
