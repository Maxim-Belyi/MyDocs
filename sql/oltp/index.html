OLTP: Строковые (Row-Oriented) базы данных

OLTP (Online Transaction Processing) — это класс задач, для которых предназначены традиционные реляционные базы данных (PostgreSQL, MySQL), т.е. обработка большого числа коротких транзакций в реальном времени (регистрация пользователя, создание заказа, обновление баланса).

Если OLAP — это про анализ (посчитай все заказы за год), то OLTP — это про операции (создай один конкретный заказ сейчас).

1. Строковое хранение (Row-Oriented Storage)​

Суть: Данные одной строки хранятся на диске последовательно, рядом друг с другом.

Представьте таблицу users:

idnameageemail
1Alice30alice@email.com
2Bob25bob@email.com

Как это выглядит на диске (упрощённо):

[1, "Alice", 30, "alice@email.com"] [2, "Bob", 25, "bob@email.com"] ...

Данные одной строки лежат рядом. Чтобы прочитать строку id=1 целиком — читаем один непрерывный блок данных. Это очень быстро.

Почему это идеально для OLTP?​

OLTP-запросы работают с конкретными строками (записями):

  • SELECT * FROM users WHERE id = 1 — читаем одну строку.
  • UPDATE orders SET status = 'paid' WHERE id = 42 — обновляем одну строку.
  • INSERT INTO users VALUES (...) — пишем одну строку.

При строковом хранении все нужные данные уже находятся рядом на диске. Одно чтение блока — и мы получили всё.

В чём проблема при аналитике?​

Допустим, нужно посчитать средний возраст всех пользователей:

SELECT AVG(age) FROM users;

Базе нужна только колонка age. Но при строковом хранении приходится читать с диска все строки целиком (id, name, age, email), хотя нужен только один столбец, это огромная лишняя работа — именно поэтому строковые БД плохо подходят для аналитики с таблицами в миллиарды строк.

2. Как OLTP обеспечивает скорость​

Write-Ahead Log (WAL)​

Это механизм, делающий OLTP-базы надёжными. Перед тем как изменить данные на диске, БД сначала записывает намерение в специальный лог (журнал).

  • Как это работает:
    1. Запрос UPDATE balance SET amount = 100 WHERE id = 1 приходит в БД.
    2. Изменение сначала записывается в WAL (быстро, последовательно).
    3. Только потом (возможно, позже, в фоне) данные обновляются в основном файле.
  • Зачем? Если сервер упал между шагами 2 и 3, при рестарте БД прочитает WAL и восстановит данные. Буква D в ACID (Durability) гарантируется именно через WAL.
  • Бонус: WAL используется также для репликации — slave-серверы читают WAL мастера и повторяют его изменения у себя.

MVCC (Multi-Version Concurrency Control)​

Механизм, который позволяет множеству транзакций работать параллельно без полной блокировки таблицы.

Идея: Вместо того чтобы блокировать строку при чтении, БД создаёт снимок (snapshot) данных на момент начала транзакции.

  • Транзакция A читает строку (видит версию v1).
  • Транзакция B в это же время обновляет ту же строку (создаёт версию v2).
  • Транзакция A продолжает видеть v1 — её никто не мешает.

Интервью-совет: Именно MVCC позволяет PostgreSQL реализовывать Repeatable Read без блокировки строк на чтение. Читатели не блокируют писателей, а писатели не блокируют читателей.

3. Индексы в OLTP​

Все OLTP запросы работают по схеме "найди конкретную строку", для этого нужны индексы.

B-Tree (Balanced Tree) — стандартный индекс для OLTP:

  • Поддерживает поиск по равенству (= 42) и по диапазону (BETWEEN 10 AND 100).
  • Поиск за $O(\log n)$. Для 1 миллиарда строк — около 30 операций сравнения.
  • При INSERT/UPDATE/DELETE индекс перестраивается — это цена за скорость чтения.

Другие типы индексов в PostgreSQL:

ТипКогда использовать
B-TreeРавенство, диапазоны, сортировка. Используется по умолчанию.
HashТолько строгое равенство (= value). Быстрее B-Tree для этого случая.
GINПолнотекстовый поиск (tsvector), поиск в JSON, массивах.
GiSTГеопространственные данные (PostGIS), диапазоны.

4. Масштабирование OLTP: Вертикальное vs Горизонтальное​

OLTP базы плохо масштабируются горизонтально (в ширину) из-за необходимости поддерживать ACID-транзакции между несколькими серверами.

Вертикальное масштабирование (Scale Up)​

Просто покупаем более мощный сервер (больше RAM, быстрее CPU, NVMe SSD). Самый простой путь, но ограничен физически.

Репликация (Read Replicas)​

Создаём копии (реплики) БД. Все записи идут на основной сервер (Primary), а чтения — на реплики (Secondary).

  • Плюс: Снимает нагрузку по чтению.
  • Минус: Реплики могут чуть отставать (репликационный лаг). Для записи всё равно один сервер — узкое место.

Шардирование (Sharding)​

Данные физически разделяются по нескольким серверам. Например, пользователи с id % 2 == 0 — на сервер A, остальные — на сервер B.

  • Плюс: Масштабируется горизонтально и по записи.
  • Минус: JOIN-ы между шардами становятся очень дорогими или вовсе невозможными, требует серьёзного проектирования.

Вывод: Если данные не влезают в один сервер и нужна горизонтальная масштабируемость — рассмотрите OLAP (колоночные) или NoSQL. OLTP правит там, где нужны транзакции и быстрый доступ к конкретным строкам.

5. Когда выбирать OLTP (строковую БД)?​

Выбираем PostgreSQL / MySQL, если:

  • Работаем с транзакциями: платежи, заказы, перевод денег.
  • Нужна строгая консистентность данных (ACID).
  • Запросы точечные: найти/изменить конкретный объект по ID.
  • Данных не более нескольких терабайт.

НЕ подходит, если:

  • Нужно агрегировать миллиарды строк (SUM, AVG, GROUP BY по всей таблице).
  • Аналитика и BI-отчёты с тяжёлыми запросами.
  • Объём данных исчисляется десятками терабайт и нужен горизонтальный скейл по записи.

➡️ Для аналитики используйте OLAP (колоночные БД), например ClickHouse.
