# Реляционные базы данных (SQL)

**Реляционная база данных (RDBMS)** — это тип базы данных, который хранит информацию в виде связанных двумерных таблиц (строк и столбцов). Слово «реляционная» происходит от английского *relation* (отношение), что в математике означает таблицу.

### Главные концепции:
*   **Схема (Schema):** Жестко заданная структура данных. Вы должны заранее объявить столбцы и их типы данных (число, строка, дата). Вы не можете просто добавить произвольный JSON-документ (хотя современные БД, вроде Postgres, это умеют, но это выход за рамки чистого реляционного подхода).
*   **Строки (Records/Tuples):** Каждая строка — это отдельная сущность (например, один конкретный пользователь).
*   **Столбцы (Attributes):** Свойства сущности (Имя, Возраст, Email).
*   **Ключи (Keys):** 
    *   *Primary Key (Первичный ключ):* Уникальный идентификатор строки (обычно ID).
    *   *Foreign Key (Внешний ключ):* Ссылка на первичный ключ в другой таблице. Именно они создают **связи (relations)** между таблицами.

## 1. Транзакции и ACID 

 Транзакция — это набор операций с базой данных, объединенных в один логический блок.

Чтобы транзакции работали надежно, БД должна соответствовать принципам **ACID**:

1.  **A - Atomicity (Атомарность):** «Всё или ничего». Транзакция либо выполняется полностью, либо не выполняется вообще. Если вы переводите деньги, и на этапе зачисления система упала, списание тоже отменится (Rollback).
2.  **C - Consistency (Согласованность):** Транзакция переводит БД из одного корректного состояния в другое. Все ограничения (constraints), внешние ключи и проверки типов должны соблюдаться. Нельзя записать строку (string) в столбец для чисел.
3.  **I - Isolation (Изолированность):** Параллельные транзакции не должны влиять друг на друга. Если два пользователя одновременно покупают один и тот же билет, БД должна обработать их так, будто они шли по очереди. 
4.  **D - Durability (Долговечность):** Если БД ответила пользователю «Успех», данные не будут потеряны даже при отключении электричества. Это достигается за счет записи логов на жесткий диск (Write-Ahead Logging / WAL).

## 2. Индексы (Ускоряем чтение)

Если вы ищете пользователя по имени без индекса `SELECT * FROM users WHERE name = 'Ivan'`, БД придется прочитать **каждую строку** в таблице (Full Table Scan), что займет $O(n)$ времени.

**Индекс** — это отдельная структура данных (аналог оглавления в книге), которая хранит отсортированные значения столбца и указатели на оригинальные строки.

*   **Как работает под капотом:** Большинство индексов используют структуру дерева **B-Tree (B+ Tree)**. Поиск по такому дереву занимает $O(\log n)$ времени. Это значит, что поиск среди 1 миллиона строк займет всего около 20 операций.
*   **Primary Key:** Первичный ключ *всегда* автоматически создает индекс.
*   **Составной индекс:** Индекс, построенный по нескольким колонкам (например, `Имя + Фамилия`). *Важное правило:* он работает только слева направо. Если искать только по Фамилии, индекс по `Имя + Фамилия` не сработает.

> **Интервью-совет:** Часто спрашивают: *"Почему бы не повесить индексы на все колонки?"*. 
> **Ответ:** Индексы сильно ускоряют **ЧТЕНИЕ**, но замедляют **ЗАПИСЬ** (Insert/Update/Delete). При каждом добавлении новой строки БД вынуждена не только писать в таблицу, но и перестраивать все B-Tree деревья для всех индексов. К тому же, индексы занимают много места на диске.

## 3. Уровни изоляции транзакций 

Изолированность (буква 'I' в ACID) бывает разной силы. Чем она сильнее, тем безопаснее данные, но тем медленнее работает БД (из-за блокировок). 

Проблемы (аномалии), которые могут возникнуть при параллельных транзакциях:
*   **Dirty Read (Грязное чтение):** Чтение данных, которые другая транзакция еще не закоммитила (и может отменить).
*   **Non-Repeatable Read (Неповторяющееся чтение):** Внутри одной транзакции вы читаете одну и ту же строку дважды, и данные меняются (кто-то другой успел их обновить).
*   **Phantom Read (Фантомное чтение):** Внутри одной транзакции вы делаете выборку `SELECT ... WHERE age > 20` дважды. Во второй раз строк становится больше (кто-то успел добавить новых пользователей).

**4 уровня изоляции (от слабого к сильному):**
1.  **Read Uncommitted:** Самый слабый. Позволяет "грязное чтение". Почти не используется.
2.  **Read Committed:** (По умолчанию в PostgreSQL). Гарантирует, что вы читаете только закоммиченные данные. Защищает от грязного чтения, но допускает неповторяющееся и фантомное.
3.  **Repeatable Read:** (По умолчанию в MySQL). Замораживает состояние строк на момент начала транзакции. Защищает от грязного и неповторяющегося чтения.
4.  **Serializable:** Самый строгий и медленный. Транзакции выполняются так, будто они идут строго друг за другом в один поток. Защищает от всех аномалий, включая фантомы.

## 4. Шпаргалка по JOIN-ам

JOIN объединяет данные из двух таблиц на основе общего столбца (Foreign Key).

*   **INNER JOIN:** Оставляет только те строки, которые есть **в обеих** таблицах. (Пересечение кругов Эйлера).
*   **LEFT JOIN (Самый частый):** Берет ВСЕ строки из левой (первой) таблицы и ищет совпадения в правой. Если совпадений нет, подставляет `NULL`.
*   **RIGHT JOIN:** Зеркально LEFT JOIN (берет всё из правой таблицы). На практике используется редко.
*   **FULL (OUTER) JOIN:** Берет ВСЕ строки из обеих таблиц. Если совпадений нет — подставляет `NULL` в пустые места.

## 5. Когда использовать реляционную БД?

**Выбираем SQL (Postgres, MySQL), если:**
*   Структура ваших данных четкая и редко меняется (например, финансовые транзакции, склад, пользователи).
*   Вам критически важны принципы ACID (нельзя потерять ни цента).
*   Данные сильно связаны между собой (нужно делать много JOIN-ов).

**Выбираем NoSQL (MongoDB, Redis, Cassandra), если:**
*   Данные не имеют строгой схемы (например, логи, карточки товаров с абсолютно разными характеристиками).
*   Требуется колоссальная скорость записи или чтения (Redis).
*   Нужно дешево и горизонтально масштабировать базу (распределять на множество серверов), так как классические реляционные БД плохо масштабируются горизонтально по записи.
