# OLAP: Колоночные (Column-Oriented) базы данных

**OLAP (Online Analytical Processing)** — класс задач для **анализа** больших объёмов исторических данных, в отличие от OLTP (работа с конкретными строками), OLAP агрегирует миллиарды строк и отвечает на вопросы:

*   «Какова выручка по каждому региону за 3 года?»
*   «Какой процент пользователей перешёл с шага 2 на шаг 3 воронки?»

> Типичные потребители: аналитики, BI-системы (Grafana, Superset, Metabase).

---

## 1. Колоночное хранение (Column-Oriented Storage)

**Суть:** Данные одного **столбца** хранятся на диске **рядом**.

Таблица `sales` с 1 миллиардом строк:

| id | product | category | amount | date |
|----|---------|----------|--------|------|
| 1  | iPhone  | Phone    | 90000  | 2024-01-01 |
| 2  | Galaxy  | Phone    | 75000  | 2024-01-02 |

**Строковое хранение (OLTP — PostgreSQL):**
```
[1, "iPhone", "Phone", 90000, "2024-01-01"] [2, "Galaxy", "Phone", 75000, "2024-01-02"]
```

**Колоночное хранение (OLAP — ClickHouse):**
```
[id]:       1, 2, 3 ...
[amount]:   90000, 75000, 65000 ...
[category]: "Phone", "Phone", "Laptop" ...
```

### Почему это даёт выигрыш?

Запрос: `SELECT SUM(amount) FROM sales WHERE category = 'Phone'`

*   **PostgreSQL:** читает **все 5 колонок** каждой строки — огромный I/O.
*   **ClickHouse:** читает только **2 колонки** — `category` и `amount`. Остальные 3 не трогает.

При широких таблицах (50+ колонок) выигрыш достигает **10-50x**.

---

## 2. Сжатие данных

Когда все значения одного столбца лежат рядом — их можно **очень эффективно сжимать**.

Колонка `category` с миллиардом строк:
```
"Phone", "Phone", "Phone", "Laptop", "Phone", "Laptop", "Laptop", ...
```

Это данные с **низкой кардинальностью**. Алгоритм **RLE** (Run-Length Encoding) сожмёт:
```
"Phone" × 3, "Laptop" × 1, "Phone" × 1, "Laptop" × 2, ...
```

Типичная степень сжатия в ClickHouse: **5:1 — 10:1**. Данные в 100 ГБ займут 10-20 ГБ.

| Алгоритм | Когда применяется |
|---|---|
| **LZ4** | По умолчанию. Баланс скорости и сжатия. |
| **ZSTD** | Лучше сжимает, чуть медленнее. |
| **Delta + LZ4** | Числовые последовательности, временные ряды. |

## 3. Векторизованное выполнение запросов

Обычный подход — обрабатывать строки по одной:
```go
for _, row := range rows {
    if row.Category == "Phone" {
        sum += row.Amount
    }
}
```

Колоночные БД обрабатывают данные **батчами (векторами)** — блоками по 8192 значений:
```
// Псевдокод "под капотом" ClickHouse
amounts  := read_column("amount",   rows=0..8191)
category := read_column("category", rows=0..8191)
sum      += SIMD_sum(amounts WHERE category == "Phone")
```

Современные CPU имеют **SIMD-инструкции** (AVX2), которые применяют одну операцию к 4-16 числам за такт. Это позволяет ClickHouse обрабатывать **миллиарды строк в секунду**.

---

## 4. Разреженный индекс (Sparse Index)

В OLTP по каждой строке есть точный B-Tree индекс. В OLAP это не нужно.

ClickHouse хранит данные **отсортированными** по первичному ключу и создаёт индекс **на каждые 8192 строк** (один гранул). Это позволяет пропускать нерелевантные блоки — не читать 900 гранул, если нужны только 5.

---

## 5. Типичная архитектура: OLTP + OLAP

В реальных системах обе парадигмы работают **совместно**:

```
Приложение
    │
    ▼ (запись транзакций)
PostgreSQL (OLTP) — продуктовая БД
    │
    │ (стриминг через Kafka / Debezium)
    ▼
ClickHouse (OLAP) — аналитическая БД
    │
    ▼
Grafana / Metabase / Superset
```

PostgreSQL хранит актуальное состояние. ClickHouse — исторические данные для тяжёлых аналитических запросов.

---

## 6. Когда выбирать OLAP?

**Выбираем ClickHouse / BigQuery, если:**
*   Аналитические агрегации по большим объёмам данных.
*   Данные пишутся пакетами (batch), а не по одной строке.
*   Таблицы широкие, но в запросах нужно 3-5 колонок из 50.

**НЕ подходит, если:**
*   Нужны транзакции (`BEGIN / COMMIT / ROLLBACK`).
*   Частое обновление отдельных строк (`UPDATE SET name = ...`).
*   Точечный доступ по ID (`SELECT * WHERE id = 42`).

---

### Шпаргалка: Row-Oriented vs Column-Oriented

| Характеристика | Строковые (PostgreSQL) | Колоночные (ClickHouse) |
| :--- | :--- | :--- |
| **Хранение** | Строки рядом | Колонки рядом |
| **Лучший запрос** | `SELECT * WHERE id = N` | `SELECT SUM(col) GROUP BY ...` |
| **Сжатие** | Слабое (~2:1) | Сильное (~10:1) |
| **Транзакции (ACID)** | Полная поддержка | Ограниченная |
| **Запись** | Одна строка, быстро | Пакетами (батчами) |
| **Индексы** | B-Tree (точные) | Разреженные (диапазонные) |
| **Когда** | Продуктовая БД | Аналитика, метрики, логи |
