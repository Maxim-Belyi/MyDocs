# RabbitMQ

## 1. Архитектура: Exchange, Queue, Binding

Ключевое отличие RabbitMQ от других брокеров — сообщение **никогда не попадает в очередь напрямую**. Оно всегда проходит через **Exchange (Обменник)**.

```
Producer → Exchange → [Binding/Routing] → Queue → Consumer
```

*   **Producer (Издатель):** Отправляет сообщение в Exchange с **Routing Key** (строка-метка, например, `order.created`).
*   **Exchange (Обменник):** Получает сообщение и решает, в какую очередь его направить, на основе **Bindings** (правил маршрутизации).
*   **Queue (Очередь):** Буфер для хранения сообщений до момента, пока Consumer их не заберёт.
*   **Consumer (Потребитель):** Подключается к очереди и получает сообщения.

### Подтверждение обработки (ACK)

Это критически важный механизм надёжности:
1.  Consumer получил сообщение из очереди.
2.  Consumer обработал сообщение (например, отправил email).
3.  Consumer отправляет **ACK (Acknowledgement)** в RabbitMQ.
4.  **Только после ACK** RabbitMQ удаляет сообщение из очереди навсегда.

Если Consumer упал до отправки ACK — RabbitMQ автоматически вернёт сообщение в очередь и доставит его другому Consumer.

> ACK — это механизм гарантии **At-least-once delivery**. Помните об идемпотентности вашего Consumer!

## 2. Типы Exchange

### Direct Exchange

Сообщение идёт в очередь, **Routing Key которой точно совпадает** с Routing Key сообщения.

```
Producer → Exchange (type: direct)
    routing_key="error"  → Queue "logs-error"
    routing_key="info"   → Queue "logs-info"
```

*Применение:* Разделение задач по типу (обработка ошибок vs информационных логов).

### Fanout Exchange (Широковещание)

**Игнорирует Routing Key** — отправляет копию сообщения **во все** привязанные очереди.

```
Producer → Exchange (type: fanout)
    → Queue "service-A" (получит копию)
    → Queue "service-B" (получит копию)
    → Queue "service-C" (получит копию)
```

*Применение:* Push-уведомления, инвалидация кэша в нескольких сервисах, рассылка событий.

### Topic Exchange (Паттерны)

Маршрутизация по **шаблону** Routing Key. Шаблон может содержать:
*   `*` — заменяет **одно** слово
*   `#` — заменяет **ноль или более** слов

```
Producer → Exchange (type: topic)
    routing_key="order.created"     → очередь с binding "order.*"    ✓
    routing_key="order.paid.stripe" → очередь с binding "order.#"    ✓
    routing_key="user.registered"   → очередь с binding "order.*"    ✗
```

*Применение:* Сложная маршрутизация событий в микросервисной архитектуре. Самый гибкий тип.

### Headers Exchange (Редко)

Маршрутизация по **заголовкам** сообщения (headers), а не по Routing Key. На практике используется редко из-за сложности настройки.

## 3. Dead Letter Queue (DLQ): Что делать с «мёртвыми» сообщениями?

**Dead Letter Queue (DLQ)** — очередь для сообщений, которые **не удалось обработать**.

Сообщение попадает в DLQ, если:
*   Consumer отправил **NACK (Negative Acknowledgement)** с флагом `requeue=false`.
*   Истёк **TTL (Time To Live)** сообщения в очереди.
*   Очередь переполнена (превышен `x-max-length`).

```
Main Queue → Consumer (ошибка при обработке)
                │
                └─ NACK (requeue=false)
                         │
                         ▼
                   Dead Letter Exchange
                         │
                         ▼
                   Dead Letter Queue (DLQ)
                         │
                         ▼
             (логирование / ручная обработка / алерт)
```

> **Это обязательный паттерн** в production-системах. Без DLQ «ядовитые» сообщения (которые всегда вызывают ошибку) могут попасть в бесконечный цикл ретраев и заблокировать всю очередь.

## 4. Надёжность: Durable Queues и Persistent Messages

По умолчанию RabbitMQ хранит данные **в памяти**. При рестарте сервера всё потеряется!

Чтобы сообщения пережили рестарт, нужно настроить **два уровня** надёжности:

1.  **Durable Queue (Устойчивая очередь):** При создании очереди указываем `durable=true`. RabbitMQ запомнит саму очередь (её метаданные) после рестарта.

2.  **Persistent Message (Устойчивое сообщение):** При отправке указываем `delivery_mode=2`. RabbitMQ запишет сообщение на **диск** (а не только в RAM).

```go
// Пример на Go (библиотека amqp091-go)
err = ch.Publish(
    "my-exchange",
    "order.created",
    false,
    false,
    amqp.Publishing{
        DeliveryMode: amqp.Persistent, // delivery_mode = 2
        ContentType:  "application/json",
        Body:         body,
    },
)
```

> **Важно:** Оба условия должны выполняться одновременно. Нет смысла делать Durable Queue, если сообщения не Persistent — при рестарте очередь восстановится пустой.

## 5. Масштабирование: Prefetch и конкурентные Consumer-ы

### Проблема: один медленный Consumer

Если Consumer обрабатывает 1 сообщение/сек, а Producer пишет 100/сек — очередь будет расти.

**Решение:** Запустить **несколько экземпляров Consumer** (горизонтальное масштабирование). RabbitMQ автоматически распределит сообщения между ними в режиме **Round-Robin**.

### Prefetch Count (QoS)

По умолчанию RabbitMQ отправит Consumer все сообщения сразу (как только они появятся). Это опасно — если Consumer получит 10 000 сообщений, закешировав их в RAM, и упадёт — все они вернутся в очередь.

`Prefetch count` ограничивает количество **неподтверждённых сообщений** у одного Consumer:

```go
err = ch.Qos(
    10,    // prefetch count: не более 10 необработанных сообщений у consumer
    0,     // prefetch size
    false, // global
)
```

## 6. Паттерны использования

### Work Queue (Очередь задач) — самый частый
Один Producer пишет задачи, несколько Consumer-ов их разбирают. Используется для CPU-интенсивных фоновых задач:
```
[Order Service] → Queue "image-processing" → [Worker 1]
                                            → [Worker 2]
                                            → [Worker 3]
```

### Publish/Subscribe через Fanout
Один Producer, множество Consumer-ов получают **копию каждого сообщения**. Аналог WebSocket broadcast для микросервисов.

### Request/Reply (RPC over RabbitMQ)
Синхронный RPC поверх асинхронного брокера. Producer отправляет сообщение и ждёт ответа в специальной очереди `reply_to`. Используется редко — если нужен RPC, лучше взять gRPC.

## 7. Порты RabbitMQ

| Порт | Протокол | Для чего |
| :--- | :--- | :--- |
| **5672** | AMQP | Основной порт. Через него приложение подключается и шлёт/получает сообщения. |
| **5671** | AMQP over TLS | Зашифрованная версия порта 5672. |
| **15672** | HTTP | Web UI (Management Plugin) — браузерная панель управления. |
| **25672** | Erlang distribution | Внутренняя коммуникация между нодами кластера RabbitMQ. |

**Почему 5672?** Это официально зарегистрированный порт стандарта AMQP в реестре IANA — не исторически сложившийся, а специально выделенный под протокол.

**В `docker-compose.yml`** обычно открывают:
```yaml
ports:
  - "5672:5672"   # для приложения
  - "15672:15672" # для Web UI (только в dev)
```

### Шпаргалка: Типы Exchange

| Тип | Маршрутизация | Применение |
| :--- | :--- | :--- |
| **Direct** | Точное совпадение Routing Key | Разделение задач по типу |
| **Fanout** | Всем (игнорирует ключ) | Push-уведомления, broadcast |
| **Topic** | По шаблону (`*`, `#`) | Гибкая маршрутизация событий |
| **Headers** | По заголовкам сообщения | Экзотические сценарии |
